الفصل الرابع: إزاي تعمل Infrastructure تكون reusable باستخدام الـ Terraform Modules

في آخر الفصل التالت، أنت عملت deploy للـ architecture اللي شفناها في شكل 4-1.

 

ده شغال تمام وزي الفل كأول environment ليك، بس أنت غالبًا بتحتاج على الأقل اتنين environments: واحدة للـ testing الداخلي بتاع فريقك (ودي اللي بنسميها staging)، وواحدة اللي الـ users الحقيقيين بيستخدموها (ودي اللي بنسميها production)، زي ما هو واضح في شكل 4-2.

المفروض إن الاتنين environments دول يكونوا شبه بعض بالظبط، بس أنت ممكن تشغل عدد servers أقل شوية أو حجمهم أصغر في الـ staging عشان توفر فلوس.

 

 

إزاي تضيف الـ production environment دي من غير ما تضطر تعمل copy and paste لكل الكود من الـ staging؟ يعني مثلاً، إزاي تتجنب إنك تعمل copy and paste لكل الكود اللي في stage/services/webserver-cluster وتحطه في prod/services/webserver-cluster، وكل الكود اللي في stage/data-stores/mysql وتحطه في prod/data-stores/mysql؟

في أي لغة برمجة general-purpose زي Ruby مثلاً، لو عندك نفس الكود معموله copy and paste في كذا مكان، كنت ممكن تحط الكود ده جوه function وتعيد استخدام الـ function دي في كل حتة:

  puts "Hello, World"
end

في Terraform بقى، أنت ممكن تحط الكود بتاعك جوه Terraform module وتعيد استخدام الـ module ده في أماكن كتير في الكود بتاعك. وبدل ما يبقى عندك نفس الكود معموله copy and paste في الـ staging والـ production environments، هتقدر تخلي الاتنين environments يعيدوا استخدام الكود من نفس الـ module، زي ما هو واضح في شكل 4-3.

 

الموضوع ده big deal جداً. الـ Modules هي العنصر الأساسي عشان تكتب كود Terraform يكونreusable وسهل إنك تعمل فيه maintenance ، وقابل للـ testing بمجرد ما تبدأ تستخدمها، مفيش رجوع. هتبدأ تبني كل حاجة كـ module، وتعمل library كاملة من الـ modules عشان تشاركها جوه الشركة بتاعتك، وتستخدم modules اللي بتلاقيها online، وهتبدأ تفكر في الـ infrastructure بتاعتك كلها على إنها مجموعة من الـ modules اللي ممكن تعيد استخدامها.

في الفصل ده، هوريك إزاي تعمل create وتستخدم Terraform modules عن طريق إننا هنتكلم عن النقط دي:

  • أساسيات الـ Module (Module basics)

  • مدخلات الـ Module (Module inputs)

  • الـ Locals في الـ Module (Module locals)

  • مخرجات الـ Module (Module outputs)

  • مشاكل ممكن تقابلك مع الـ Module (Module gotchas)

  • إصدارات الـ Module (Module versioning)

 

أساسيات الـ Module

الـ Terraform module بسيط جدًا: أي مجموعة من الـ Terraform configuration files في folder هي module.

كل الـ configurations اللي كتبتها لحد دلوقتي كانت modules من الناحية techincal، ، لإنك عملتلها deploy مباشرةً: لو عملت run لـ apply مباشرة على module، ده بيتسمى root module.

عشان تشوف إيه اللي الـ modules تقدر تعمله بجد، أنت محتاج تعمل create لـ reusable module، وده module معمول عشان تستخدمه جوه modules تانية او root module

كمثال، خلينا نحول الكود اللي في stage/services/webserver-cluster، واللي بيشمل Auto Scaling Group (ASG)، و Application Load Balancer (ALB)، و security groups، و resources تانية كتير، لـ reusable module.

كأول خطوة، اعمل run لـ terraform destroy في stage/services/webserver-cluster عشان تمسح أي resources كنت عملتها قبل كده. بعد كده، اعمل create لـ top-level folder جديد اسمه modules، وانقل كل الـ files من stage/services/webserver-cluster لـ modules/services/webserver-cluster. المفروض إنك في الآخر هيكون عندك folder structure شكله عامل زي شكل 4-4.

افتح ملف الـ main.tf اللي في modules/services/webserver-cluster، وامسح

الـ provider definition. الـ Providers المفروض يتعملها configure بس في الـ root modules ومش في الـ reusable modules (هتتعلم أكتر بكتير عن الشغل مع الـ providers في الفصل السابع).

Tip

provider definition

xxxxxxxxxx
provider "aws" {
region = "us-east-2" }

... أي إعدادات تانية للـ AWS provider فى الفصل السابع هتشوف ازاي هتتعامل مع multiprovider

 

 

أنت دلوقتي تقدر تستخدم الـ module ده في الـ staging environment. وده الـ syntax عشان تستخدم module:

xxxxxxxxxx
module "<NAME>" {
  source = "<SOURCE>"
}
[CONFIG ...]

حيث إن NAME ده identifier تقدر تستخدمه في كود الـ Terraform كله عشان تشاور بيه على الـ module ده (مثلاً webserver_cluster) تقدر تسمي اي اسم برحتك، و SOURCE هو الـ path اللي كود الـ module ده موجود فيه (مثلاً modules/services/webserver-cluster)، والـ CONFIG ده بيكون عبارة عن arguments خاصة بالـ module ده بالذات هنشوف ازاي قدام شوية ازاي ادخل argument

كمثال، أنت ممكن تعمل create لـ file جديد في stage/services/webserver-cluster/main.tf وتستخدم الـ webserver-cluster module جواه بالشكل ده:

xxxxxxxxxx
provider "aws" {
  region = "us-east-2"
}


module "webserver_cluster" {
  source = "../../../modules/services/webserver-cluster"
}

وبعد كده تقدر تعيد استخدام نفس الـ module بالظبط في الـ production environment عن طريق إنك تعمل create لـ file جديد اسمه prod/services/webserver-cluster/main.tf بالمحتويات دي:

xxxxxxxxxx
provider "aws" {
  region = "us-east-2"
}


module "webserver_cluster" {
  source = "../../../modules/services/webserver-cluster"
}

وده كده، code reuse في كذا environment بحجم تكرار بسيط جداً. لاحظ إن كل مرة بتضيف module للـ Terraform configurations بتاعتك أو بتعدّل في الـ source parameter بتاع module، بتحتاج تعمل run لـ init command قبل ما تعمل run لـ plan أو apply:

xxxxxxxxxx
$ terraform init
Initializing modules...
- webserver_cluster in ../../../modules/services/webserver-cluster
Initializing the backend...
Initializing provider plugins...
Terraform has been successfully initialized!

 

دلوقتي أنت شفت كل الإمكانيات اللي الـ init command بيعملها: هو بيعمل install للـ providers، و configure للـ backends بتاعتك، و download للـ modules، كل ده في command واحد سهل.

قبل ما تعمل run لـ apply command على الكود ده، خد بالك إن فيه مشكلة في الـ webserver-cluster module: كل الأسماء اللي جواه hardcoded (يعني مكتوبة ومثبتة يدويًا). بمعنى تاني، اسم الـ security groups، والـ ALB، و resources تانية كتير، كلها hardcoded. فلو استخدمت الـ module ده أكتر من مرة في نفس الـ AWS account، هيجيلك name conflict errors. حتى التفاصيل بتاعت إزاي تقرا الـ database state برضه hardcoded، لإن ملف الـ main.tf اللي أنت عملتله copy في modules/services/webserver-cluster بيستخدم terraform_remote_state data source عشان يعرف الـ database address والـ port، والـ terraform_remote_state ده hardcoded إنه يبص على الـ staging environment.

Note

Hardcoded معناها إن قيمة معينة أو إعدادت معينه مكتوبة بشكل ثابت ومباشر جوه الكود نفسه، ومينفعش يتغير إلا لو فتحت الكود وعدلته يدويًا.

عشان تصلح المشاكل دي، أنت محتاج تضيف configurable inputs (مدخلات قابلة للتعديل) للـ webserver-cluster module عشان يقدر يتصرف بشكل مختلف في environments مختلفة.

الـ Module Inputs

عشان تخلي أي function قابلة للتعديل (configurable) في أي لغة برمجة general-purpose زي Ruby مثلاً، ممكن تضيفلها input parameters:

xxxxxxxxxx
# A function with two input parameters
def example_function(param1, param2)
  puts "Hello, #{param1} #{param2}"
end


# Pass two input parameters to the function
example_function("foo", "bar")

في Terraform، الـ modules ممكن يكون ليها input parameters برضه. عشان تعرفهم، هتستخدم طريقة أنت عارفها بالفعل: الـ input variables. افتح الملف modules/services/webserver-cluster/variables.tf وضيف تلاتة input variables جداد:

xxxxxxxxxx
variable "cluster_name" {
  description = "The name to use for all the cluster resources"
  type        = string
}


variable "db_remote_state_bucket" {
  description = "The name of the S3 bucket for the database's remote state"
  type        = string
}


variable "db_remote_state_key" {
  description = "The path for the database's remote state in S3"
  type        = string
}

بعد كده، ادخل على ملف modules/services/webserver-cluster/main.tf، واستخدم var.cluster_name بدل الأسماء الـ hardcoded (يعني بدل "terraform-asg-example" مثلاً). كمثال، دي الطريقة اللي تعمل بيها كده للـ ALB security group:

xxxxxxxxxx
resource "aws_security_group" "alb" {
  name = "${var.cluster_name}-alb"
  ingress {
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }
  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }
}

لاحظ إزاي الـ name parameter بقى قيمته ${var.cluster_name}-alb. أنت هتحتاج تعمل تغيير مشابه للـ aws_security_group resource التاني (يعني تديله اسم "${var.cluster_name}-instance" مثلاً)، وللـ aws_alb resource، وللـ tag section اللي في الـ aws_autoscaling_group resource.

المفروض كمان تعمل update للـ terraform_remote_state data source عشان يستخدم الـ db_remote_state_bucket والـ db_remote_state_key كـ bucket و key parameter على التوالي، عشان تضمن إنك بتقرا الـ state file من الـ environment الصح:

xxxxxxxxxx
data "terraform_remote_state" "db" {
  backend = "s3"
  config = {
    bucket = var.db_remote_state_bucket
    key    = var.db_remote_state_key
    region = "us-east-2"
  }
}

دلوقتي، في الـ staging environment، في ملف stage/services/webserver-cluster/main.tf، تقدر تحدد الـ input variables الجديدة دي على حسب الـ environment كده:

xxxxxxxxxx
module "webserver_cluster" {
  source = "../../../modules/services/webserver-cluster"
  cluster_name           = "webservers-stage"
  db_remote_state_bucket = "(YOUR_BUCKET_NAME)"
  db_remote_state_key    = "stage/data-stores/mysql/terraform.tfstate"
}

والمفروض برضه تحدد الـ variables دي في الـ production environment في ملف prod/services/webserver-cluster/main.tf، بس بقيم مختلفة تتناسب مع الـ environment دي:

xxxxxxxxxx
module "webserver_cluster" {
  source = "../../../modules/services/webserver-cluster"
  cluster_name           = "webservers-prod"
  db_remote_state_bucket = "(YOUR_BUCKET_NAME)"
  db_remote_state_key    = "prod/data-stores/mysql/terraform.tfstate"
}

ملاحظة الـ production database لسه مش موجودة في الحقيقة. كـ exercise ليك، أنا بسيبلك إنك تضيف production database شبه الـ staging بالظبط.

زي ما أنت شايف، أنت بتحدد الـ input variables للـ module باستخدام نفس الـ syntax بتاع تحديد الـ arguments للـ resource. الـ input variables هي الـ API بتاع الـ module، وهي اللي بتتحكم في إزاي الـ module هيتصرف في الـ environments المختلفة.

لحد دلوقتي، أنت ضفت input variables لاسم الـ cluster والـ remote state بتاع الـ database، بس أنت ممكن تكون عايز تخلي parameters تانية قابلة للتعديل في الـ module بتاعك برضه. كمثال، في الـ staging، ممكن تكون عايز تشغل web server cluster صغير عشان توفر فلوس، لكن في الـ production، ممكن تكون عايز تشغل cluster أكبر عشان يتعامل مع حجم كبير من الـ traffic. عشان تعمل كده، ممكن تضيف تلاتة input variables كمان لـ modules/services/webserver-cluster/variables.tf:

xxxxxxxxxx
variable "instance_type" {
  description = "The type of EC2 Instances to run (e.g. t2.micro)"
  type        = string
}


variable "min_size" {
  description = "The minimum number of EC2 Instances in the ASG"
  type        = number
}


variable "max_size" {
  description = "The maximum number of EC2 Instances in the ASG"
  type        = number
}

بعد كده، اعمل update للـ launch configuration في modules/services/webserver-cluster/main.tf عشان تحدد الـ instance_type parameter بتاعه لـ input variable الجديد var.instance_type:

xxxxxxxxxx
resource "aws_launch_configuration" "example" {
  image_id        = "ami-0fb653ca2d3203ac1"
  instance_type   = var.instance_type # التغيير هنا
  security_groups = [aws_security_group.instance.id]
  user_data = templatefile("user-data.sh", {
    server_port = var.server_port
    db_address  = data.terraform_remote_state.db.outputs.address
    db_port     = data.terraform_remote_state.db.outputs.port
  })
  # Required when using a launch configuration with an auto scaling group.
  lifecycle {
    create_before_destroy = true
  }
}

وبنفس الطريقة، المفروض تعمل update للـ ASG definition في نفس الملف عشان تحدد الـ min_size و max_size parameters لـ input variables الجديدة var.min_size و var.max_size على التوالي:

xxxxxxxxxx
resource "aws_autoscaling_group" "example" {
  launch_configuration = aws_launch_configuration.example.name
  vpc_zone_identifier  = data.aws_subnets.default.ids
  target_group_arns    = [aws_lb_target_group.asg.arn]
  health_check_type    = "ELB"
  min_size             = var.min_size # التغيير هنا
  max_size             = var.max_size # والتغيير هنا
  tag {
    key                 = "Name"
    value               = var.cluster_name
    propagate_at_launch = true
  }
}

دلوقتي، في الـ staging environment (stage/services/webserver-cluster/main.tf)، تقدر تخلي الـ cluster صغير ومش مكلف عن طريق إنك تحدد الـ instance_type لـ "t2.micro" و الـ min_size و max_size لـ 2:

xxxxxxxxxx
module "webserver_cluster" {
  source = "../../../modules/services/webserver-cluster"
  cluster_name           = "webservers-stage"
  db_remote_state_bucket = "(YOUR_BUCKET_NAME)"
  db_remote_state_key    = "stage/data-stores/mysql/terraform.tfstate"
  instance_type          = "t2.micro"
  min_size               = 2
  max_size               = 2
}

على الناحية التانية، في الـ production environment، ممكن تستخدم instance_type أكبر فيه CPU و memory أكتر، زي m4.large (خد بالك إن الـ Instance type ده مش جزء من الـ AWS Free Tier، فلو أنت بتستخدم ده مجرد للتعلم ومش عايز تتشحن، خليك مع "t2.micro" للـ instance_type)، وممكن تحدد الـ max_size لـ 10 عشان تسمح للـ cluster إنه يصغر أو يكبر على حسب الـ load (متخافش، الـ cluster هيقوم بـاتنين Instances في الأول):

xxxxxxxxxx
module "webserver_cluster" {
  source = "../../../modules/services/webserver-cluster"
  cluster_name           = "webservers-prod"
  db_remote_state_bucket = "(YOUR_BUCKET_NAME)"
  db_remote_state_key    = "prod/data-stores/mysql/terraform.tfstate"
  instance_type          = "m4.large"
  min_size               = 2
  max_size               = 10
}

 

الـ Locals في الـ Module

استخدام الـ input variables عشان تحدد الـ inputs بتاعت الـ module بتاعك حاجة ممتازة، بس إيه اللي هيحصل لو أنت محتاج طريقة عشان تعرف variable جوه الـ module بتاعك عشان تعمل بيه شوية calculations (حسابات وسيطة)، أو عشان تخلي الكود بتاعك DRY (مختصر ومفيهوش تكرار)، بس في نفس الوقت مش عايز تخلي الـ variable ده يظهر كـ configurable input للمستخدمين؟ يعني ميكونواش variables inputs ليك كمستخدام

كمثال، الـ load balancer اللي في الـ webserver-cluster module في الملف modules/services/webserver-cluster/main.tf بيسمع على port 80، اللي هو الـ default port للـ HTTP. رقم الـ port ده حاليًا معموله copy and paste في أكتر من مكان، بما فيهم

الـ load balancer listener:

xxxxxxxxxx
resource "aws_lb_listener" "http" {
  load_balancer_arn = aws_lb.example.arn
  port              = 80
  protocol          = "HTTP"
  # By default, return a simple 404 page
  default_action {
    type = "fixed-response"
    fixed_response {
      content_type = "text/plain"
      message_body = "404: page not found"
      status_code  = 404
    }
  }
}

وكمان في الـ load balancer security group:

xxxxxxxxxx
resource "aws_security_group" "alb" {
  name = "${var.cluster_name}-alb"
  ingress {
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }
  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }
}

القيم اللي في الـ security group، بما في ذلك الـ CIDR block بتاع "كل الـ IPs" اللي هو 0.0.0.0/0، وقيمة "أي port" اللي هي 0، وقيمة "أي protocol" اللي هي "-1"، برضه معمولها copy and paste في كذا مكان جوه الـ module. إن الـ magical values (القيم اللي مش واضحة معناها) دي تبقى hardcoded في كذا مكان بيخلي الكود أصعب في القراءة والـ maintenance.

يعني لو حبيت تغير القيم دي هي اه مش input varables بس لو حبيت تغيرها فى module ذات نفسه

طب ليه مخليش مستخدام module يغيرها لانه ممكن يسبب اخطاء كتير بذات لو Module بتاعك كبير ومعقد احنا بنكلم عليك كا الشخص اللي بيأسس module

أنت ممكن تاخد القيم دي وتحطها في input variables، بس ساعتها users الـ module بتاعك هيقدروا يغيروا القيم دي (بالغلط)، وده ممكن متكونش عايزه. بدل ما تستخدم input variables، ممكن تعرف القيم دي كـ local values في locals block:

xxxxxxxxxx
locals {
  http_port    = 80
  any_port     = 0
  any_protocol = "-1"
  tcp_protocol = "tcp"
  all_ips      = ["0.0.0.0/0"]
}

الـ Local values بتسمحلك تدي اسم لأي Terraform expression وتستخدم الاسم ده في كل الـ module. الأسماء دي بتبقى ظاهرة بس جوه الـ module ده، فمش بيكون ليها أي تأثير على modules تانية، ومتقدرش تغير القيم دي من بره الـ module. عشان تقرا قيمة الـ local، بتحتاج تستخدم local reference، اللي بتستخدم الـ syntax ده:

xxxxxxxxxx
local.<NAME>

استخدم الـ syntax ده عشان تعمل update للـ port parameter بتاع الـ load-balancer listener بتاعك:

xxxxxxxxxx
resource "aws_lb_listener" "http" {
  load_balancer_arn = aws_lb.example.arn
  port              = local.http_port # التغيير هنا
  protocol          = "HTTP"
  # By default, return a simple 404 page
  default_action {
    type = "fixed-response"
    fixed_response {
      content_type = "text/plain"
      message_body = "404: page not found"
      status_code  = 404
    }
  }
}

وبنفس الطريقة، اعمل update لكل الـ parameters تقريبًا في الـ security groups جوه الـ module، بما في ذلك الـ load-balancer security group:

xxxxxxxxxx
resource "aws_security_group" "alb" {

  ingress {
    from_port   = local.http_port    # التغيير هنا

    cidr_blocks = local.all_ips      # وهنا
  }
  egress {
    from_port   = local.any_port     # وهنا
    to_port     = local.any_port     # وهنا
    protocol    = local.any_protocol # وهنا
    cidr_blocks = local.all_ips      # وهنا
  }
}

مخرجات الـ Module (Module Outputs)

واحدة من الـ features القوية جداً بتاعت الـ ASGs هي إنك ممكن تعملها configure عشان تزود أو تقلل عدد الـ servers اللي شغالة عندك بناءً على الـ load (الضغط). طريقة من طرق عمل كده هي استخدام scheduled action، ودي ممكن تغير حجم الـ cluster في وقت محدد خلال اليوم. كمثال، لو الـ traffic على الـ cluster بتاعك أعلى بكتير خلال ساعات الشغل العادية، ممكن تستخدم scheduled action عشان تزود عدد الـ servers الساعة 9 الصبح وتقلله الساعة 5 بعد الضهر.

لو أنت عرّفت الـ scheduled action ده جوه الـ webserver-cluster module، ده هيطبق على الـ staging والـ production الاتنين. ولإنك مش محتاج تعمل النوع ده من الـ scaling في الـ staging environment بتاعتك، ففي الوقت الحالي، ممكن تعرف الـ auto scaling schedule مباشرةً في الـ production configurations (في الفصل الخامس، هتشوف إزاي تعمل define للـ resources بشكل conditionally، وده اللي هيخليك تنقل الـ scheduled action جوه الـ webserver-cluster module).

عشان تعرف scheduled action، ضيف الـ aws_autoscaling_schedule resources الاتنين دول لملف prod/services/webserver-cluster/main.tf:

xxxxxxxxxx
resource "aws_autoscaling_schedule" "scale_out_during_business_hours" {
  scheduled_action_name = "scale-out-during-business-hours"
  min_size              = 2
  max_size              = 10
  desired_capacity      = 10
  recurrence            = "0 9 * * *"
}


resource "aws_autoscaling_schedule" "scale_in_at_night" {
  scheduled_action_name = "scale-in-at-night"
  min_size              = 2
  max_size              = 10
  desired_capacity      = 2
  recurrence            = "0 17 * * *"
}

الكود ده بيستخدم aws_autoscaling_schedule resource واحد عشان يزود عدد الـ servers لـ 10 خلال ساعات الصبح (الـ recurrence parameter بيستخدم cron syntax، يعني "0 9 * * *" معناها "الساعة 9 الصبح كل يوم")، و aws_autoscaling_schedule resource تاني عشان يقلل عدد الـ servers بالليل ("0 17 * * *" معناها "الساعة 5 بعد الضهر كل يوم").

بس، الـ aws_autoscaling_schedule في الاستخدامين دول ناقصه parameter مطلوب اسمه autoscaling_group_name، واللي بيحدد اسم الـ ASG. الـ ASG نفسه متعرف جوه الـ webserver-cluster module، فإزاي تقدر توصل لاسمه؟ في أي لغة برمجة general-purpose زي Ruby مثلاً، الـ functions ممكن ترجع قيم:

xxxxxxxxxx
# A function that returns a value
def example_function(param1, param2)
  return "Hello, #{param1} #{param2}"
end


# Call the function and get the return value
return_value = example_function("foo", "bar")

في Terraform، الـ module برضه ممكن يرجع قيم. ومرة تانية، أنت بتعمل كده باستخدام طريقة أنت عارفها بالفعل: الـ output variables. أنت ممكن تضيف اسم الـ ASG كـ output variable في modules/services/webserver-cluster/outputs.tf بالشكل ده:

xxxxxxxxxx
output "asg_name" {
  value       = aws_autoscaling_group.example.name
  description = "The name of the Auto Scaling Group"
}

تقدر توصل للـ output variables بتاعت الـ module باستخدام الـ syntax ده:

xxxxxxxxxx
module.<MODULE_NAME>.<OUTPUT_NAME>

كمثال:

xxxxxxxxxx
module.frontend.asg_name

في الملف prod/services/webserver-cluster/main.tf، ممكن تستخدم الـ syntax ده عشان تحدد قيمة الـ autoscaling_group_name parameter في كل واحد من الـ aws_autoscaling_schedule resources:

xxxxxxxxxx
resource "aws_autoscaling_schedule" "scale_out_during_business_hours" {
  scheduled_action_name    = "scale-out-during-business-hours"
  min_size                 = 2
  max_size                 = 10
  desired_capacity         = 10
  recurrence               = "0 9 * * *"
  autoscaling_group_name   = module.webserver_cluster.asg_name
}


resource "aws_autoscaling_schedule" "scale_in_at_night" {
  scheduled_action_name    = "scale-in-at-night"
  min_size                 = 2
  max_size                 = 10
  desired_capacity         = 2
  recurrence               = "0 17 * * *"
  autoscaling_group_name   = module.webserver_cluster.asg_name
}

أنت ممكن تكون عايز تعرض output تاني في الـ webserver-cluster module: الـ DNS name بتاع الـ ALB، عشان تعرف الـ URL اللي تختبر بيه لما الـ cluster يتعمله deploy. عشان تعمل كده، مرة تانية بتضيف output variable في modules/services/webserver-cluster/outputs.tf:

xxxxxxxxxx
output "alb_dns_name" {
  value       = aws_lb.example.dns_name
  description = "The domain name of the load balancer"
}

وبعد كده ممكن "تمرر" الـ output ده في stage/services/webserver-cluster/outputs.tf و prod/services/webserver-cluster/outputs.tf بالشكل ده:

xxxxxxxxxx
output "alb_dns_name" {
  value       = module.webserver_cluster.alb_dns_name
  description = "The domain name of the load balancer"
}

الـ web server cluster بتاعك تقريبًا جاهز للـ deploy. الشيء الوحيد اللي فاضل هو إننا ناخد كام gotcha (مشاكل ممكن تقابلنا) في الاعتبار.

 

 

مشاكل ممكن تقابلك مع الـ Module (Module Gotchas)

لما تيجي تعمل create لـ modules، خد بالك كويس من المشاكل اللي ممكن تقابلك ودي اسمها gotchas. أهم اتنين دلوقتي هما:

  • File Paths

  • الـ Inline Blocks

(File Paths)

في الفصل التالت، أنت نقلت الـ User Data script بتاع الـ web server cluster لـ external file اسمه user-data.sh، واستخدمت الـ built-in function اللي اسمها templatefile عشان تقرا الملف ده من الـ disk.

الـ catch أو المشكلة اللي ممكن تحصل مع templatefile function هي إن الـ filepath (مسار الملف) اللي بتستخدمه لازم يكون relative path — يعني مش مسار كامل من أول الـ root بتاع الجهاز بتاعك، لإن الكود بتاع Terraform بتاعك ممكن يشتغل على أجهزة مختلفة كتير، وكل جهاز ليه ترتيب disk مختلف.

طيب، الـ relative path ده بيبقى نسبي لإيه بالظبط؟

بشكل default (افتراضي)، Terraform بيفسر الـ path على إنه نسبي بالنسبة للـ current working directory (الفولدر اللي أنت واقف فيه حاليًا وبتشغل منه كوماند terraform).

  • ده بيشتغل كويس لو أنت بتستخدم templatefile function في ملف Terraform configuration اللي موجود في نفس الفولدر اللي بتعمل منه run لأمر terraform apply (ودي الحالة اللي بنسميها root module في الـ deployment ده). يعني لو الـ main.tf بتاعك هو اللي فيه templatefile وهو اللي بتشغل منه apply، يبقى user-data.sh هيلاقي ملف user-data.sh اللي جنبه.

  • لكن، ده مش هيشتغل لو أنت بتستخدم templatefile جوه module متعرف في folder منفصل (وده اللي بنسميه reusable module)، وانت بتعمل apply من root module تاني خالص.

تخيل معايا السيناريو ده:

عندك الـ folder structure ده:

xxxxxxxxxx
my-project/
├── modules/
│   └── webserver-module/   # <--- ده الـ Reusable Module بتاعك
│       ├── main.tf         
│       └── user-data.sh    
│
└── environments/
    └── staging/           
        └── main.tf         # (هنا main.tf بينادي على الـ)

لما أنت بتعمل run لـ terraform apply من جوه environments/staging/، الـ current working directory بتاعتك بتبقى environments/staging/.

لو جوه modules/webserver-module/main.tf كتبت: user_data = templatefile("user-data.sh", { ... })

Terraform جوه الـ module ده هيحاول يدور على user-data.sh فين؟ هيدور عليه relative للـ current working directory اللي أنت بتعمل منها apply، يعني هيدور في environments/staging/user-data.sh! وهو طبعًا مش موجود هناك، هو موجود جوه الـ module نفسه في modules/webserver-module/user-data.sh.

الحل: استخدام path.module

عشان نحل المشكلة دي، ممكن نستخدم expression اسمه path reference، اللي بيكون شكله path.<TYPE>. بيدعم الأنواع دي من الـ path references:

  • path.module:

    • بيرجع الـ filesystem path بتاع الـ module اللي الـ expression ده متعرف جواه.

  • path.root:

    • بيرجع الـ filesystem path بتاع الـ root module (يعني الفولدر اللي أنت بتعمل منه run لأمر terraform apply).

  • path.cwd:

    • بيرجع الـ filesystem path بتاع الـ current working directory (الفولدر اللي أنت واقف فيه حاليًا). في الاستخدام العادي لـ Terraform، ده بيكون هو هو path.root، لكن في بعض الاستخدامات المتقدمة لـ Terraform، بيتم تشغيله من directory غير الـ root module directory، وده بيخلي المسارين دول مختلفين.

بالنسبة للـ User Data script، أنت محتاج path نسبي بالنسبة للـ module نفسه، عشان كده المفروض تستخدم path.module لما بتنادي على الـ templatefile function في ملف modules/services/webserver-cluster/main.tf:

xxxxxxxxxx
user_data = templatefile("${path.module}/user-data.sh", {
  server_port = var.server_port
  db_address  = data.terraform_remote_state.db.outputs.address
  db_port     = data.terraform_remote_state.db.outputs.port
})

بكده، الـ templatefile function هتبقى دايمًا عارفة مكان user-data.sh بالظبط، بغض النظر أنت بتنادي على الـ module ده من أنهي environment أو root module.

 

الـ Inline Blocks والـ Separate Resources

في Terraform، بعض الـ resources ممكن تعملها configure أو تحدد إعداداتها بطريقتين مختلفتين. المثال الشائع هنا هو aws_security_group اللي ممكن تعرف قواعد الدخول والخروج (ingress و egress rules) فيه بطريقتين.

الطريقة الأولى: الـ Inline Blocks

  • إيه هي؟ دي arguments بتتحط مباشرةً جوه الـ resource الرئيسي نفسه. بتكون عبارة عن block بيبقى له اسم محدد، وجواه الـ parameters الخاصة بيه.

xxxxxxxxxx
resource "aws_security_group" "example" {
  name    = "my-security-group"
  vpc_id  = "vpc-xxxxxxxx"




  ingress {
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }




  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1" 
    cidr_blocks = ["0.0.0.0/0"]
  }
}

  • في المثال ده، قواعد الـ ingress والـ egress تعتبر جزء من تعريف الـ resource اللي اسمه aws_security_group.example. الـ resource ده بيعمل manage لكل قواعد الدخول والخروج دي كوحدة واحدة. لو حبيت تعدل قاعدة، لازم تعدل الـ resource الرئيسي ده نفسه.

الطريقة الثانية: الـ Separate Resources

  • إيه هي؟ دي resources منفصلة تمامًا عن الـ resource الرئيسي. كل قاعدة ingress أو egress بتكون resource لوحدها، وبتربطها بالـ Security Group الرئيسي عن طريق الـ ID بتاعه.

xxxxxxxxxx
# 1. بنعرف الـ Security Group نفسه، بدون أي قواعد داخلية
resource "aws_security_group" "example" {
  name   = "my-security-group-separate"
  vpc_id = "vpc-xxxxxxxx"
}




resource "aws_security_group_rule" "allow_http_inbound" {
  type              = "ingress"
  security_group_id = aws_security_group.example.id 
  from_port         = 80
  to_port           = 80
  protocol          = "tcp"
  cidr_blocks       = ["0.0.0.0/0"]
}




resource "aws_security_group_rule" "allow_all_outbound" {
  type              = "egress"
  security_group_id = aws_security_group.example.id  
  from_port         = 0
  to_port           = 0
  protocol          = "-1"
  cidr_blocks       = ["0.0.0.0/0"]
}

 

هنا كل قاعدة (سواء ingress أو egress) هي resource مستقل بذاته (aws_security_group_rule). ده بيخلي إدارة القواعد أكثر granular، وتقدر تضيف أو تعدل rule واحد بس من غير ما تأثر على تعريف الـ Security Group الرئيسي.

المشكلة: ليه مينفعش تخلط بين الطريقتين؟

الـ problem إنك مينفعش تستخدم الطريقتين دول مع بعض لنفس الـ resource. يعني، لو عرفت ingress rule جوه الـ aws_security_group كـ inline block (بالطريقة الأولى)، متقدرش في نفس الوقت تعرف ingress rule تاني لنفس الـ aws_security_group باستخدام separate aws_security_group_rule resource (بالطريقة الثانية).

السبب تقنيًا: Terraform لما بيشوف inline blocks جوه resource رئيسي، بيعتبر إنه هو المسؤول الكامل عن إدارة كل تلك الـ blocks. لو جيت بعد كده تعرف نفس الـ blocks كـ separate resources، Terraform هيشوف إن فيه تعارض. مين فيهم هو المسؤول عن إدارة القواعد دي؟ ده بيخلي Terraform يحاول يعمل overwrite (يمسح ويكتب فوق القديم) أو يعمل delete و recreate بشكل مستمر، مما يؤدي إلى errors وعدم استقرار.

لازم تختار يا إما كل قواعد الـ Security Group تكون inline، يا إما كلها تكون separate resources.

دمثال هيولد مشكلة 

xxxxxxxxxx
# 1. Define the AWS Provider
provider "aws" {
  region = "us-east-1" # You can change the region if needed
}


# 2. Define a simple VPC for the Security Group
resource "aws_vpc" "example_vpc" {
  cidr_block = "10.0.0.0/16"


}


# 3. Define a Security Group with an INLINE INGRESS BLOCK
resource "aws_security_group" "conflict_sg" {
  name        = "conflict-security-group"
  description = "Security Group to demonstrate rule conflict"
  vpc_id      = aws_vpc.example_vpc.id


  # --- Defining an Ingress Rule as an INLINE BLOCK ---
  ingress {
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
    description = "Allow HTTP inbound (inline)"
  }


  # Add a simple egress rule to keep the SG valid
  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1" # -1 means all protocols

  }


  tags = {
    Name = "ConflictSG"
  }
}


# 4. Attempt to add the same type of rule (Ingress) to the same Security Group
# But this time as a SEPARATE RESOURCE (aws_security_group_rule)
# --- THIS LINE WILL GENERATE THE CONFLICT ERROR ---
resource "aws_security_group_rule" "conflicting_ingress_rule_separate" {
  type              = "ingress"
  security_group_id = aws_security_group.conflict_sg.id # Referring to the same Security Group above
  from_port         = 22 # Different port to clearly show the conflict intent
  to_port           = 22
  protocol          = "tcp"
  cidr_blocks       = ["0.0.0.0/0"]
  description       = "Allow SSH inbound (separate, CONFLICTING)"
}

       on Security Group 'sg-xxxxxxxxxxxxxxxxx' (security_group_id)
       which are already managed by the 'ingress' block within the

Best Practice وليه الـ Separate Resources أحسن للموديول:

لما تكون بتبني Terraform module قابل reusable، دايمًا الأفضل إنك تفضل استخدام الـ separate resources لقواعد الـ Security Group.

ليه؟ الـ separate resources بتوفر flexibility (مرونة) أعلى بكتير:

  • التعديل من برة برة: لو الـ module بتاعك بيعمل Security Group، وكنت عامل الـ rules بتاعته inline، المستخدم اللي هيستخدم الـ module ده (اللي بينادي عليه) مش هيقدر يضيف rule زيادة للـ Security Group ده من بره الـ module. هيضطر يعدل الـ module نفسه عشان يضيف الـ rule، وده بيقلل الـ reusability وبيصعب الـ maintenance.

  • الإضافة الديناميكية: لكن لو الـ rules معرفة كـ separate resources، والـ module بتاعك بيطلع الـ ID بتاع الـ Security Group كـ output variable، ساعتها المستخدم يقدر بمنتهى السهولة يضيف aws_security_group_rule جديد في الـ root module بتاعه ويشاور على الـ ID اللي الـ module رجعه.

    • مثال: module بتاعك بيعمل Web Server وبيفتح port 80 بس. في الـ staging environment، أنت عايز تفتح port 12345 للـ testing. لو الـ rules separate، تقدر تضيف aws_security_group_rule جديد في كود الـ staging يفتح port 12345 على الـ Security Group بتاع الـ module من غير ما تعدل الـ module نفسه.

بكده، الـ separate resources بتديلك مرونة أكبر بكتير في تصميم الـ infrastructure بتاعتك وتخصيصها لـ environments أو استخدامات مختلفة بدون ما تضطر تعدل في الـ module الأساسي.

 

ودي نصيحتي ليك: لما تيجي تعمل create لـ module، المفروض دايمًا تفضل استخدام الـ separate resources .

الميزة في استخدام الـ separate resources هي إنك ممكن تضيفها في أي مكان، في حين إن الـ inline block ممكن يتضاف بس جوه الـ module اللي بيعمل الـ resource الرئيسي. عشان كده، استخدام الـ separate resources بس بيخلي الـ module بتاعك مرن أكتر وقابل للتعديل (flexible and configurable) أكتر.

كمثال، في الـ webserver-cluster module (في الملف modules/services/webserver-cluster/main.tf)، أنت كنت بتستخدم inline blocks عشان تعرف الـ ingress والـ egress rules كده:

xxxxxxxxxx
resource "aws_security_group" "alb" {
  name = "${var.cluster_name}-alb"
  ingress { 
    from_port   = local.http_port
    to_port     = local.http_port
    protocol    = local.tcp_protocol
    cidr_blocks = local.all_ips
  }
  egress { 
    from_port   = local.any_port
    to_port     = local.any_port
    protocol    = local.any_protocol
    cidr_blocks = local.all_ips
  }
}

مع الـ inline blocks دي، المستخدم اللي هيستخدم الـ module ده مالوش أي طريقة إنه يضيف ingress أو egress rules زيادة من بره الـ module. عشان تخلي الـ module بتاعك مرن أكتر، المفروض تغيره عشان يعرف نفس الـ ingress والـ egress rules دي بالظبط بس باستخدام separate aws_security_group_rule resources (اتأكد إنك تعمل كده للـ security groups الاتنين اللي في الـ module):

xxxxxxxxxx


resource "aws_security_group" "alb" {
  name = "${var.cluster_name}-alb"
  #  inline ingress/egress blocks
}




resource "aws_security_group_rule" "allow_http_inbound" {
  type              = "ingress"
  security_group_id = aws_security_group.alb.id 
  from_port         = local.http_port
  to_port           = local.http_port
  protocol          = local.tcp_protocol
  cidr_blocks       = local.all_ips
}




resource "aws_security_group_rule" "allow_all_outbound" {
  type              = "egress"
  security_group_id = aws_security_group.alb.id 
  from_port         = local.any_port
  to_port           = local.any_port
  protocol          = local.any_protocol
  cidr_blocks       = local.all_ips
}

المفروض كمان تعمل export للـ ID بتاع الـ aws_security_group كـ output variable في الملف modules/services/webserver-cluster/outputs.tf:

xxxxxxxxxx
output "alb_security_group_id" {
  value       = aws_security_group.alb.id
  description = "The ID of the Security Group attached to the load balancer"
}

دلوقتي، لو احتجت تفتح port زيادة في الـ staging environment بس (مثلاً للـ testing)، تقدر تعمل ده عن طريق إنك تضيف aws_security_group_rule resource لملف stage/services/webserver-cluster/main.tf (الـ root module بتاع الـ staging):

xxxxxxxxxx
module "webserver_cluster" {
  source = "../../../modules/services/webserver-cluster"


}




resource "aws_security_group_rule" "allow_testing_inbound" {
  type              = "ingress"
  security_group_id = module.webserver_cluster.alb_security_group_id #here
  from_port         = 12345
  to_port           = 12345
  protocol          = "tcp"
  cidr_blocks       = ["0.0.0.0/0"]
}

لو كنت عرفت حتى ingress أو egress rule واحد كـ inline block، الكود ده مكنش هيشتغل. خد بالك إن نفس النوع ده من المشاكل بيأثر على عدد من Terraform resources، زي دول:

  • aws_security_group و aws_security_group_rule

  • aws_route_table و aws_route

  • aws_network_acl و aws_network_acl_rule

في النقطة دي، أنت أخيرًا جاهز تعمل deploy للـ web server cluster بتاعك في الـ staging والـ production الاتنين. اعمل run لـ terraform apply كالعادة، واستمتع باستخدام نسختين منفصلتين من الـ infrastructure بتاعتك.

Important

الـ NETWORK ISOLATION

الأمثلة اللي في الفصل ده بتعمل اتنين environments معزولين عن بعض في كود Terraform بتاعك، وكمان معزولين من ناحية إن كل واحد ليه load balancers و servers و databases منفصلة. بس هما مش معزولين على مستوى الـ network. عشان نخلي كل الأمثلة في الكتاب ده بسيطة، كل الـ resources بتتعملها deploy في نفس الـ VPC ده معناه إن أي server في الـ staging environment يقدر يكلم أي server في الـ production environment ، والعكس صحيح.

في real-world usage إنك تشغل الاتنين environments في VPC واحدة بيعرضك لنوعين من المخاطر. أولاً، أي غلطة في environment واحدة ممكن تأثر على التانية. كمثال، لو بتعمل تغييرات في الـ staging وبالغلط لخبطت الـ configuration بتاعت الـ route tables ، كل الـ routing في الـ production ممكن يتأثر هو كمان. ثانياً، لو attacker (مهاجم) قدر يوصل لـ environment واحدة، يبقى كده وصل للتانية كمان. لو بتعمل تغييرات سريعة في الـ staging وبالغلط سبت port مفتوح، أي hacker قدر يدخل هيكون عنده access مش بس للـ staging data بتاعتك، لأ ده كمان للـ production data.

عشان كده، بعيدًا عن الأمثلة البسيطة والـ experiments، المفروض تشغل كل environment في VPC منفصلة. في الحقيقة، عشان تتأكد أكتر، ممكن تشغل كل environment في AWS account منفصلة تماماً.

 

 

إصدارات الـ Module (Module Versioning)

لو الـ staging environment و الـ production environment بتوعك الاتنين بيشاوروا على نفس الـ module folder، أول ما هتعمل أي change (تغيير) في الـ folder ده، هيأثر على الاتنين environments في الـ deployment اللي جاي على طول. النوع ده من الـ coupling (الارتباط/الربط) بيخلي إنك تعمل test لـ change في الـ staging أصعب بكتير من غير ما يكون فيه أي فرصة إنه يأثر على الـ production.

الطريقة الأفضل هي إنك تعمل create لـ versioned modules (موديولات ليها إصدارات) عشان تقدر تستخدم version (إصدار) في الـ staging (زي v0.0.2) و version مختلف في الـ production (زي v0.0.1)، زي ما هو واضح في شكل 4-5.

في كل الـ module examples اللي شفتها لحد دلوقتي، كل ما كنت بتستخدم module، كنت بتحدد الـ source parameter بتاع الـ module لـ local filepath . بالإضافة للـ file paths، Terraform بيدعم أنواع تانية من الـ module sources ، زي Git URLs، و Mercurial URLs، و arbitrary HTTP URLs.

 

 

إدارة إصدارات الـ Module (Module Versioning)

الطريقة الأفضل هي إنك تعمل create لـ versioned modules (موديولات ليها إصدارات) عشان تقدر تستخدم version في الـ staging (زي v0.0.2) و version مختلف في الـ production (زي v0.0.1).

إزاي تحقق Module Versioning ده؟

أسهل طريقة عشان تعمل versioned module هي إنك تحط الكود بتاع الـ module في Git repository منفصل، وتحدد الـ source parameter لـ URL بتاع الـ repository ده. ده معناه إن كود الـ Terraform بتاعك هيكون متوزع على (على الأقل) اتنين repositories (مستودعات Git):

  • الـ Repository الأول: modules الـ repo ده وظيفته إنه تحط فيه الـ reusable modules . كل folder جواه بيعرف module معين بيحدد جزء من الـ infrastructure بتاعتك. ده بيحتوي على ملفات Terraform configuration لكل module (زي main.tf, variables.tf, outputs.tf). الـ versions بتاعت الـ modules دي بتتعمل عن طريق Git tags (زي v0.0.1, v0.0.2).

  • الـ Repository الثاني: live الـ repo ده وظيفته إنه يعرف الـ live infrastructure (البنية التحتية اللي شغالة بالفعل) اللي أنت مشغلها في كل environment (زي stage, prod, mgmt, وغيرهم). الـ repo ده بيكون جواه فولدرات لكل environment، وكل environment folder بيكون فيه الـ Terraform configuration اللي بتنادي على الـ modules من الـ modules repository، وتحدد version معين لكل module.

الـ folder structure المحدث لكود الـ Terraform بتاعك، بعد ما قسمت الـ repositories بالطريقة دي، دلوقتي بقى شكله عامل زي شكل 4-6 (الصورة اللي أنت حاططها في السؤال).

عشان تعمل setup للـ folder structure ده، أنت هتحتاج الأول تنقل الـ folders اللي هما stage، و prod، و global لـ folder اسمه live. بعد كده، اعمل configure لـ folders اللي هما live و modules كـ separate Git repositories. وده مثال إزاي تعمل كده للـ modules folder:

$ git init
$ git add .
$ git commit -m "Initial commit of modules repo"
$ git remote add origin "(URL OF REMOTE GIT REPOSITORY)"
$ git push origin main

أنت ممكن كمان تضيف tag للـ modules repo عشان تستخدمه كـ version number . لو أنت بتستخدم GitHub، ممكن تستخدم واجهة GitHub UI عشان تعمل create لـ release، وده هيعمل create لـ tag في under the hood` فى الخلفية.

لو أنت مش بتستخدم GitHub، ممكن تستخدم Git CLI:

xxxxxxxxxx

Important

 

الشرطتين دول // (Double-Slash) ليهم وظيفة محددة جداً في الـ Terraform module source لما بيكون المصدر Git repository.

فايدتهم الأساسية هي إنهم بيحددوا لـ Terraform إن الـ module code اللي أنت عايز تستخدمه مش موجود في الـ root بتاع الـ Git repository كله.

لكن هو موجود في subdirectory (فولدر فرعي) جوه الـ repository ده.

الجزء اللي قبل الـ // (يعني github.com/foo/modules) ده بيكون الـ URL بتاع الـ Git repository نفسه.

والجزء اللي بعد الـ // (يعني services/webserver-cluster) ده بيكون الـ path اللي جوه الـ repository ده بيشاور على الـ folder اللي فيه الكود بتاع الـ module بالظبط.

ده بيساعد لما يكون عندك repository واحد فيه كذا module منفصل، بدل ما تعمل repository لكل module لوحده. من غير الـ // والـ path اللي بعده، Terraform هيحاول يدور على الـ module بتاعك في الـ root بتاع الـ repository ومش هيلاقيه.

لو عايز تجرب الـ versioned modules من غير ما تتعامل مع Git repos، ممكن تستخدم module من الـ code examples GitHub repo بتاع الكتاب ده:

xxxxxxxxxx
source = "github.com/brikis98/terraform-up-and-running-code//
code/terraform/04-terraform-module/module-example/modules/
services/webserver-cluster?ref=v0.3.0"

الـ ref parameter بيسمحلك تحدد Git commit معين عن طريق الـ sha1 hash بتاعه، أو branch name، أو، زي المثال ده، Git tag معين. أنا بشكل عام بنصح باستخدام Git tags كـ version numbers للـ modules. أسماء الـ branches مش stable (مش ثابتة)، لإنك دايمًا بتاخد آخر commit على الـ branch، وده ممكن يتغير كل مرة بتعمل run لـ init command، والـ sha1 hashes مش human friendly . الـ Git tags ثابتة زي الـ commit

الـ naming scheme (نظام التسمية) المفيد بشكل خاص للـ tags هو الـ semantic versioning (الترقيم الدلالي للإصدارات). ده نظام versioning بيكون شكله MAJOR.MINOR.PATCH (مثلاً 1.0.4) مع قواعد محددة لمتى المفروض تزود كل جزء من رقم الـ version. بشكل خاص، المفروض تزود الآتي:

  • الـ MAJOR version لما تعمل incompatible API changes (تغييرات في الـ API مش متوافقة مع الإصدارات اللي قبلها).

  • الـ MINOR version لما تضيف functionality (وظائف جديدة) بطريقة backward-compatible (متوافقة مع الإصدارات اللي قبلها).

  • الـ PATCH version لما تعمل backward-compatible bug fixes (تصلح أخطاء بطريقة متوافقة مع الإصدارات اللي قبلها).

الـ Semantic versioning بيديلك طريقة عشان تتواصل مع مستخدمين الـ module بتاعك إيه نوع التغييرات اللي عملتها وإيه تأثير الـ upgrade (الترقية).

ولإنك عملت update لكود Terraform بتاعك عشان يستخدم versioned module URL، أنت محتاج تدي تعليمات لـ Terraform عشان يعمل download لكود الـ module عن طريق إنك تعمل rerun (تشغل تاني) لـ terraform init:

xxxxxxxxxx
$ terraform init
Initializing modules...
Downloading git@github.com:brikis98/terraform-up-and-running-code.git?
ref=v0.3.0
for webserver_cluster...
(...)

المرة دي، هتشوف إن Terraform بيعمل download لكود الـ module من Git بدل ما يعمل download من الـ local filesystem . بعد ما كود الـ module يتعمله download، ممكن تعمل run لـ apply command كالعادة.

Warning

التأكيد على مبدأ "The Single Responsibility Principle" في الـ Modules

المبداء ده مكمل معاك في حياتك كا Devops لنصايب تانية للـcontainer و pod وتقريبا اي حاجة معدية و

دي نقطة مهمة جدًا في تصميم الـ modules وممكن تضيفها في بداية الفصل أو بعد "أساسيات الـ Module".

خلي الـ Module بتاعك يعمل حاجة واحدة بس، بس يعملها صح

وانت بتصمم الـ modules بتاعتك، خلي في بالك مبدأ مهم من هندسة البرمجيات اسمه Single Responsibility Principle. يعني إيه؟ يعني الـ module الواحد المفروض يكون مسئول عن حاجة واحدة ومنطقية.

مثال غلط: تعمل module واحد اسمه my-whole-app بيعمل VPC, subnets, database, load balancer, servers كلهم مع بعض. الـ module ده هيبقى ضخم، معقد، وصعب جدًا تعدل فيه أي حاجة من غير ما تبوظ حاجة تانية.

مثال صح: تعمل module للـ vpc، وmodule تاني للـ database، وmodule تالت للـ webserver-cluster. وبعدين في الـ root module بتاعك (زي staging أو production)، تنادي على الـ modules الصغيرة دي وتركبهم مع بعض زي الليجو.

التقسيمة دي بتخلي كل module صغير، مركز، سهل الفهم، سهل الاختبار، وسهل إعادة استخدامه في مشاريع تانية.

 

 

 

Tip

مشكلة الـ User Data الكبيرة : ليه الـ Script بيشتغل على جهازي وبيفشل على السيرفر؟

واحدة من أكتر المشاكل اللي بتلخبط الناس وهي بتستخدم Terraform مع User Data، هي إن الـ script يكون شغال تمام وزي الفل على جهازك، وأول ما تعمل apply، تلاقي الـ VM قامت بس فاضية وفيها errors غريبة. المشكلة دي سببها سوء فهم بسيط بس مهم جدًا للطريقة اللي Terraform بيتعامل بيها مع الملفات.

لازم نفرق بين مكانين وزمانين مختلفين:

  1. Configuration Environment (على جهازك):** دي البيئة اللي Terraform بيشتغل فيها. هو بيقرأ ملفاتك (.tf و .sh) اللي على الهارد ديسك بتاعك، بيجمع كل حاجة، وبيجهز أمر نهائي. path.module دي وظيفتها بس تساعد Terraform يعرف مكان الملفات دي على جهازك أنت.

  2. Runtime Environment (على الكلاود):** دي البيئة اللي جوه الـ VM اللي Terraform بينشاءها ا. الـ VM دي ليها filesystem خاص بيها، ومعزولة تمامًا عن جهازك. هي بتاخد الأمر النهائي اللي Terraform بعته وتبدأ تنفذه.

path.module بتشتغل بس في Configuration Environment. أول ما الأمر يتبعت للكلاود، path.module دي بتبقى ولا ليها أي لازمة.

السيناريو اللي بيعمل المشكلة: لما بنقسم الـ User Data Script

لنفترض إنك عشان تنظم شغلك، قررت تقسم الـ User Data Script لملفين بدل ملف واحد:

  1. user-data.sh: ده الملف الأساسي اللي بينادي على الأوامر بالترتيب.

  2. helpers.sh: ده ملف فيه كل الـ functions اللي بتعمل الشغل التقيل.

  # ...


  user_data = templatefile("${path.module}/user-data.sh", {
    server_port = var.server_port
  })
  # ...
}

المشكلة بتحصل فين بالظبط؟ (The Point of Failure)

يلا نتتبع اللي بيحصل خطوة بخطوة عشان نعرف إيه اللي بيبوظ:

1. مرحلة terraform apply (بيئة الإعداد على جهازك):

  • Terraform بيقرأ main.tf.

  • بيوصل لـ templatefile("${path.module}/user-data.sh", ...)

  • بيستخدم path.module عشان يلاقي ملف user-data.sh على جهازك. بيلاقيه عادي جدًا.

  • Terraform بيقرأ محتويات user-data.sh بس. هو مش بيفهم bash، فميعرفش إن source ./helpers.sh دي معناها إنه المفروض يروح يقرأ ملف تاني. بالنسبة له، ده مجرد سطر نصي.

  • بيجهز الأمر النهائي اللي هو محتوى ملف user-data.sh زي ما هو.

  • بيبعت النص ده لـ AWS API.

اللي AWS استلمته هو النص ده بس:

xxxxxxxxxx
#!/bin/bash
echo "Starting main script..."
source ./helpers.sh
echo "Installing web server..."
install_web_server
echo "Configuring firewall..."
configure_firewall
echo "Script finished."

2. Runtime Environment الـ VM ( في الكلاود):**

  • الـ EC2 Instance (الـ VM) بتقوم وبتاخد الـ script اللي فات ده عشان تنفذه.

  • بتنفذ echo "Starting main script...". بتشتغل تمام.

  • بتوصل للسطر ده: source ./helpers.sh.

  • هنا بتحصل المشكلة. الـ VM بتحاول تدور على ملف اسمه helpers.sh في الفولدر اللي هي فيه.

  • مش بتلاقيه. لأن الملف ده أصلاً لسه موجود على جهازك أنت في مصر، والـ VM شغالة في داتا سنتر في أمريكا مثلاً. الملف ده عمره ما اتبعَت.

  • الـ VM بتروح مسجلة Error في الـ logs بتاعتها، بيقول حاجة زي كده:

    xxxxxxxxxx
    /var/lib/cloud/instance/scripts/part-001: line 4: ./helpers.sh: No such file or directory

  • بعدها، الـ script بيحاول يكمل وينفذ install_web_server. الـ VM هترد تقول:

    xxxxxxxxxx
    /var/lib/cloud/instance/scripts/part-001: line 8: install_web_server: command not found

    لأن الـ function دي كانت متعرفة في ملف helpers.sh اللي أصلاً متحملش.

الـ deployment بتاعك هيفشل. الـ VM هتكون running لكن فاضية، مفيش أي حاجة اتسطبت عليها صح، وكل الـ logs بتاعتها هتكون مليانة errors.

الحل الصح في السيناريو ده إيه؟

لو الـ scripts بتاعتك معتمدة على بعض، لازم تضمن إنها كلها توصل للـ server. عندك كذا طريقة:

الحل الاولتدمجهم في template واحد: تخلي user-data.sh هو الـ template الرئيسي وتستخدم templatefile عشان تضمن محتوى helpers.sh جواه. الطريقة دي مش عملية لو الـ scripts كتيرة.

خلينا نشوف إزاي نعملها بالكود.

هنرجع لنفس هيكل الملفات بتاعنا:

  • main.tf

  • user-data.sh (ده هيبقى الـ template الرئيسي)

  • helpers.sh (ده الـ template الفرعي اللي عايزين ندمجه)

تعديل الكود عشان يدمج الملفات

الخطوة الأولى: تعديل الملف الرئيسي (user-data.sh)

هنعدل user-data.sh عشان يستقبل محتوى helpers.sh كـ variable جايه من Terraform.

modules/services/webserver-cluster/user-data.sh (بعد التعديل):

xxxxxxxxxx
#!/bin/bash


# =============================================================
${helpers_script_content}
# =============================================================




echo "Starting main script..."


echo "Installing web server..."
install_web_server


echo "Configuring firewall..."
configure_firewall


echo "Script finished."

إيه اللي اتغير؟

  • شيلنا السطر بتاع source ./helpers.sh.

  • حطينا مكانه variable جديد اسمه ${helpers_script_content}. الفكرة إن Terraform هو اللي هيعوض عن الـ variable ده بمحتويات ملف helpers.sh كلها.

الخطوة الثانية: تعديل main.tf عشان يعمل الشغلانة دي

دلوقتي هنعدل الـ call بتاعة templatefile في main.tf عشان تعمل الخطوات دي بالترتيب:

  1. تقرأ ملف helpers.sh وتعمله render (لو فيه variables جواه).

  2. تاخد الناتج اللي طلع من الخطوة 1.

  3. تقرأ الملف الرئيسي user-data.sh.

  4. تبعت الناتج بتاع helpers.sh كـ variable للملف الرئيسي user-data.sh عشان يدمجه.

modules/services/webserver-cluster/main.tf (بعد التعديل):

xxxxxxxxxx


resource "aws_launch_configuration" "example" {




  user_data = templatefile("${path.module}/user-data.sh", {
    server_port            = var.server_port




    helpers_script_content = templatefile("${path.module}/helpers.sh", {}) 
  })
  # ...
}

إيه اللي حصل في الكود ده؟

  • الـ templatefile الخارجية بتشتغل على user-data.sh.

  • بنباصي ليها map فيها variable اسمه helpers_script_content.

  • قيمة الـ variable ده هي ناتج تشغيل templatefile تانية على helpers.sh. القوسين {} الفاضيين معناهم إن helpers.sh معندوش أي variables محتاجين نعوض عنها.

النتيجة النهائية اللي بتوصل للـ VM

Terraform لما ينفذ الكود اللي فات ده، هيطلع string نهائي واحد مدموج فيه كل حاجة. الـ User Data script اللي هيوصل للـ EC2 instance هيبقى شكله كده:

xxxxxxxxxx
#!/bin/bash






function install_web_server() {


  echo "Web server installed."
}


function configure_firewall() {


  echo "Firewall configured."
}






echo "Starting main script..."


echo "Installing web server..."
install_web_server


echo "Configuring firewall..."
configure_firewall


echo "Script finished."

بكده، الـ VM لما تستلم الـ script ده، كل الـ functions هتكون متعرفة وجاهزة للاستخدام، ومفيش أي errors هتحصل بسبب إن ملف ناقص.

ليه الطريقة دي مش عملية لو الـ scripts كتيرة؟ لأنك هتضطر تعمل nesting (تداخل) لـ templatefile calls كتير، والكود في main.tf هيبقى شكله معقد وصعب في القراءة والصيانة. تخيل عندك 5 ملفات مساعدة، هتحتاج 5 templatefile calls جوه الـ map. عشان كده، الطريقة دي كويسة للملفات القليلة، لكن لو الموضوع كبر، الحل بتاع تنزيل الـ scripts من S3 بيكون أنضف وأكثر احترافية

تنزلهم من مكان مركزي (الطريقة الاحترافية):

ترفع كل الـ scripts بتاعتك (user-data.sh, helpers.sh, etc.) على S3 Bucket.

الـ User Data script اللي بتبعته مع Terraform يكون بسيط جداً، وظيفته الوحيدة إنه ينزل الـ scripts الحقيقية من S3 وبعدين يشغلها.

متنساش تبقى تزبط IAM Role الله يرضى عليك مش متقليش error بتطلع من كل حتة

 

xxxxxxxxxx
# في Terraform
user_data = <<-EOF
#!/bin/bash
aws s3 cp s3://my-scripts-bucket/user-data.sh /tmp/user-data.sh
aws s3 cp s3://my-scripts-bucket/helpers.sh /tmp/helpers.sh
chmod +x /tmp/user-data.sh
/tmp/user-data.sh
EOF

 

 

Important

خدها قاعدة: Inline للمعلومات الأساسية، Separate للحاجات اللي بتتغير

فكر فيها كده:

  • Inline Blocks (جوه الـ resource): استخدمها لما تكون بتعرّف الخصائص الأساسية والثابتة للـ resource واللي مش هتتغير من بره الـ module. زي مثلاً name أو vpc_id.

  • Separate Resources (بره الـ resource): استخدمها للحاجات اللي ممكن تتغير أو تتضاف عليها من بره الـ module، زي الـ security group rules أو الـ route table entries.

ده بيخلي الـ module بتاعك مرن زي "قطعة ليجو": ليه شكل أساسي ثابت، بس تقدر تركب عليه قطع إضافية من بره حسب الحاجة.

 

Tip

 

وماذا عن الـ Terraform Registry؟

بجانب استخدام Git مباشرةً، فيه طريقة تانية شائعة جداً ومحترفة لنشر واستخدام الـ modules وهي Terraform Registry.

Public Registry: ده المكان اللي بتلاقي فيه آلاف الـ modules الجاهزة من HashiCorp والـ community (زي modules لعمل VPC أو EKS cluster). الـ source بتاعها بيبقى شكله كده:

xxxxxxxxxx
source = "terraform-aws-modules/vpc/aws"
version = "3.14.0"

Private Registry: الشركات الكبيرة غالباً بتعمل Private Registry خاص بيها (باستخدام Terraform Cloud أو Artifactory مثلاً). ده بيخلي مشاركة الـ modules جوه الشركة منظمة جداً وسهلة، وبيوفر gui تقدر تشوف فيها كل الـ modules المتاحة وإصداراتها والـ documentation بتاعتها.

استخدام Registry بيخلي إدارة الـ versions واكتشاف الـ modules أسهل بكتير من التعامل مع Git URLs مباشرةً، وبيعتبر هو الـ Best Practice في البيئات الكبيرة.

لو انت فى شركة ومش عايز تدفع ممكن تستخدم Nexus والدنيا هتمشي مش هتققف وبرده فى حولول مفتوحة المصدر بس معرفهمش

 

Important

خلي الـ Documentation تتكتب لوحدها!

بما إن الـ inputs والـ outputs (مع الـ descriptions بتاعتهم) هما الـ API بتاع الـ module بتاعك، مهم جدًا إنك تعمل documentation واضحة ليهم. الطريقة اليدوية هي إنك تعمل ملف README.md وتكتب فيه جدول للـ inputs والـ outputs.

بس فيه طريقة أسهل وأكثر احترافية: استخدم أداة open-source اسمها terraform-docs. الأداة دي بتقرأ كود الـ Terraform بتاعك (ملفات variables.tf و outputs.tf) وبتولد documentation منظمة وجميلة بشكل أوتوماتيكي.

تقدر تشغلها كجزء من الـ pre-commit hook بتاعك في Git، عشان تضمن إن الـ README.md بتاع أي module يكون دايمًا up-to-date مع أي تغيير في الكود. ده بيوفر وقت وبيقلل الأخطاء وبيخلي الـ modules بتاعتك احترافية جدًا.

 

Tip

الـ Modules بتنادي على Modules (Module Composition)

قوة الـ modules الحقيقية بتظهر لما تبدأ تركبها فوق بعض. مش لازم الـ module بتاعك يكون بيحتوي على resources بس. أنت ممكن تعمل module "أعلى مستوى" وظيفته إنه ينادي على شوية modules تانية أصغر وينسق ما بينهم.

مثال: ممكن تعمل module اسمه webapp-with-database. الـ module ده نفسه مفيهوش aws_instance أو aws_db_instance، لكن جواه بينادي على:

  1. module "database" اللي هو الـ module بتاعك بتاع الـ MySQL.

  2. module "webserver" اللي هو الـ module بتاعك بتاع الـ ASG.

وهو اللي بيقوم بدور الوسيط، بياخد الـ outputs من module الداتا بيز (زي الـ endpoint) ويباصيها كـ inputs لـ module الـ webserver.

الطريقة دي بتخليك تبني abstractions أعلى وأعلى، وتخلي الكود في الـ live repo بتاعك بسيط جداً ومقروء.

 

Note

إزاي تعمل كذا نسخة من نفس الـ Module؟

لحد دلوقتي، إحنا بننادي على الـ module مرة واحدة بس في كل environment. طب لو أنت عايز تعمل deploy لـ 3 webserver clusters مختلفين في نفس الـ environment (يمكن واحد للـ API، وواحد للـ Admin Panel، وواحد للـ Website)؟

هل هتعمل copy-paste للـ module بلوك 3 مرات؟ لأ طبعًا!

زي ما الـ resources بتقبل count و for_each، الـ modules كمان بتقبلهم. دي طريقة قوية جدًا عشان تعمل deploy لنسخ متعددة من نفس الـ module بشكل ديناميكي.

xxxxxxxxxx
locals {
clusters = {
api    = { instance_type = "t3.medium", min_size = 5 }
admin  = { instance_type = "t3.small",  min_size = 1 }
}
}


module "webserver_cluster" {
for_each = local.clusters


source        = "..."
cluster_name  = "webserver-${each.key}"
instance_type = each.value.instance_type
min_size      = each.value.min_size
# ...
}

هنتكلم بالتفصيل عن count و for_each والـ loops في الفصل الجاي، بس خلي الفكرة دي في بالك لإنها بترفع مستوى الـ automation بتاعك لحتة تانية خالص.
