1. Git-Flow :

تخيل معايا إنك بتبني عمارة ضخمة، مينفعش كل الصنايعية يشتغلوا في نفس الشقة في نفس الوقت، صح؟ لازم نظام. الـ Git-Flow ده بقى هو النظام ده بس للبرمجة. هو طريقة محترمة كده بتقولك إزاي تدير الـ branches بتاعتك، وكل branch دوره إيه بالظبط في الفيلم الكبير بتاع المشروع.

  • الـ branches الأساسية في مسرح عمليات Git-Flow:

    • main (أو كانوا زمان بيسموه master، بس main أشيك وألطف): ده يا سيدي اعتبره "النسخة النهائية" اللي بتتسلم للعميل. ده الكود اللي شغال زي الفل عند الناس، الـ production code النضيف اللي مفيش فيه غلطة (أو المفروض يعني!). محدش بيقرب من الـ branch ده أو يعمل commit عليه مباشرة كده وخلاص. لا يا حبيبي، ده خط أحمر! الـ main ده ميتلمسش إلا لما تكون بتطلع release جديد، يعني نسخة جديدة من البرنامج بتاعك جاهزة ومختبرة ومية فل وعشرة. كل commit على الـ main ده بيكون بمثابة ختم النسخة الرسمية للمنتج، ولازم يتعمله tag برقم الـ version ده عشان تبقى عارف كل نسخة طلعت إيه بالظبط (مثلاً: v1.0, v1.0.1, v2.0). تخيلها كده زي شريط الافتتاح اللي بيتقص لما المشروع يخلص.

    • develop: ده بقى "الورشة الكبيرة" أو "المطبخ الرئيسي" اللي بيتطبخ فيه كل الشغل الجديد. ده الـ branch اللي بيكون فيه الشغل اللي لسه تحت التطوير (active development). أي feature جديدة، أي تعديل، أي تجربة عايز تعملها، بتبدأ شغلك من هنا (أو بمعنى أصح، بتاخد منه branch فرعي) ولما تخلص بترجع تصب شغلك فيه تاني. الـ develop ده المفروض يكون نضيف إلى حد كبير، يعني الكود اللي فيه شغال ومستقر، بس ممكن يكون فيه حاجات لسه متعملهاش release رسمي. ده الـ branch اللي بيستقبل كل الـ features الجديدة بعد ما تخلص.

  • feature branches: تخيل إنك عايز تضيف زرار جديد في البرنامج، أو تعمل صفحة تسجيل دخول بشكل مختلف، أو أي (feature) جديدة. هنا بقى بتاخد branch جديد من الـ develop وتسميه اسم ليه علاقة بالـ feature اللي بتعملها (مثلاً feature/user-login-system أو feature/add-dark-mode). الاسم الواضح ده مهم عشان تبقى عارف كل branch بيعمل إيه. بتشتغل بقى على الـ feature branch ده براحتك خالص، تعدل وتضيف وتجرب من غير ما تقلق إنك تبوظ شغل حد تاني أو تبوظ الـ develop الأساسي. لما تخلص الـ feature بتاعتك خالص، وتعملها test كويس وتتأكد إنها شغالة مية مية ومبتعملش مشاكل، بتروح عاملها merge تاني مع الـ develop branch. كأنك كنت واخد حتة من develop شكلتها براحتك، ولما خلصت رجعتها للعجينة الكبيرة.

 

release branches : لما يتجمع عندك شوية features حلوين كده على الـ develop branch، والفريق يقرر إن "يلا بينا يا رجالة، عايزين نطلع version جديدة للناس"، هنا بقى بيجي دور الـ release branch. بتاخد branch جديد من الـ develop (اللي فيه كل الـ features الجديدة) وتسميه باسم الـ release اللي هتطلعه (مثلاً release/v1.2 أو release/version-2.1.0). الـ release branch ده عامل زي "فلتر تنقية نهائي" أو "البوابة الأخيرة" قبل ما الحاجة تروح للـ main (الـ production). في الـ branch ده، فريق الـ QA (بتوع اختبار الجودة) بيمسكوه يفلوه، يعملوا عليه اختبارات مكثفة جداً. لو طلع أي bugs بسيطة أو تعديلات طفيفة أخيرة، بتتصلح مباشرة على الـ release branch ده. مهم جداً: مفيش أي features جديدة بتخش الـ release branch ده خالص! هو بس لآخر تلميعات وإصلاح bugs خفيفة وتجهيزات الـ release. كأنك بتجهز العربية قبل ما تسلمها للعميل، بتغسلها وتلمعها وتتأكد إن كل حاجة في مكانها.

 

  • hotfix branches : دي بقى للطوارئ اللي بجد! "ألحق يا ريس، الموقع وقع!" أو "العميل الفلاني مش عارف يدفع وفيه bug خطير في الـ production!". لو حصلت كارثة زي دي في الكود اللي على الـ main (يعني اللي العملاء بيستخدموه فعلاً)، ومحتاجة تتصلح فوراً فوراً، هنا بتعمل branch جديد مباشرة من الـ main (خد بالك: من الـ main مش من الـ develop، لأن المشكلة في الـ production ولازم تتصلح على نفس الكود اللي فيه المشكلة). بتسمي الـ branch ده باسم الـ hotfix (مثلاً hotfix/critical-payment-bug أو hotfix/urgent-login-fix). بتصلح المشكلة دي بأسرع ما يمكن، ولما تخلص وتتأكد إنها اتصلحت، بتعمل merge للـ hotfix branch ده على main (عشان تصلح المشكلة عند العملاء فوراً) وكمان بتعمل merge على develop (عشان المشكلة دي متظهرش تاني في الـ features الجديدة أو الـ releases الجاية اللي بتطبخ على develop). وطبعاً، بعد ما تعمل merge على main، بتعمل tag جديد للـ version دي (مثلاً لو كنت على v1.2.0، الـ hotfix يخليها v1.2.1).

 

  • دورة حياة الشغل (Life-cycle) في Git-Flow عامله إزاي:

    1. البداية من develop: المطور عايز يعمل feature جديدة (مثلاً "إضافة تعليقات على المقالات").

    2. يخلق feature branch: ياخد branch من develop ويسميه feature/article-comments.

    3. شغل على الـ feature: يشتغل براحته على الـ feature/article-comments، يعمل commits زي ما هو عايز.

    4. انتهاء الـ feature: لما يخلص الخاصية دي، يعملها merge تاني على develop. كده develop بقى فيه الخاصية الجديدة دي بالإضافة لخواص تانية خلصها زمايله.

    5. قرار عمل release: يتجمع كام feature كده على develop، ويقولوا خلاص هنطلع version جديدة (مثلاً v2.0).

    6. يخلق release branch: ياخدوا branch من develop ويسموه release/v2.0.

    7. اختبارات وتلميع: الـ QA يشتغلوا على release/v2.0، يصلحوا أي bugs بسيطة، يعملوا آخر تعديلات. (مفيش features جديدة هنا!)

    8. التمام:

      • Merge على main: يتعمله merge على main. كده الكود الجديد وصل للـ production.

      • Tag على main: يتعمل tag للـ commit الأخير على main باسم v2.0.

      • Merge التعديلات من release على develop: أي تعديلات أو إصلاحات bugs حصلت على release/v2.0 أثناء الاختبار، لازم يتعملها merge تاني على develop، عشان develop يبقى فيه كل حاجة محدثة.

    9. لو حصلت طوارئ (hotfix):

      • مشكلة في production (main): فجأة اكتشفنا bug خطير في v2.0 اللي على main.

      • بتنشاء hotfix branch من main: ناخد branch من main ونسميه hotfix/fix-that-nasty-bug.

      • الإصلاح الفوري: نصلح الـ bug بسرعة البرق.

      • Merge الـ hotfix على main: نعمل merge لـ hotfix/fix-that-nasty-bug على main.

      • Tag جديد على main: نعمل tag جديد، مثلاً v2.0.1.

      • Merge الـ hotfix على develop: نعمل merge لـ hotfix/fix-that-nasty-bug على develop (وعلى الـ release branch لو كان فيه واحد شغال حالياً ومفتوح) عشان الـ bug ده ميظهرش تاني في الشغل الجديد.

  • مميزات Git-Flow الحلوة (Pros):

    • بتنظم الدنيا في الفرق الكبيرة: لما يكون فيه كذا فريق شغالين أو عدد المطورين كبير، الطريقة دي بتخلي كل واحد عارف هو بيعمل إيه بالظبط، والمسؤوليات متوزعة صح. بتمنع الدنيا تسيع من بعضها.

    • بتعرف تدير كذا version من المنتج في نفس الوقت ببراعة: يعني ممكن تكون بتعمل صيانة وبتصلح bugs في version قديم (عن طريق hotfix) وفي نفس الوقت فريق تاني شغال على features جديدة خالص لـ version أحدث على develop.

    • مسؤوليات واضحة لكل branch كأنها بطاقة شخصية: مفيش لخبطة. الـ main ده بتاع الـ production وبس. الـ develop هو ملعب التطوير الرئيسي. الـ feature لكل خاصية على حدة. الـ release لتجهيز الإصدار النهائي. والـ hotfix لإطفاء الحرايق.

    • سهل ترجع لأي version قديم طلع production بالـ tags: دي ميزة جبارة. الـ tags بتخليك عامل زي "سجل يا تاريخ". لو حصلت مشكلة في version جديد طلعته، ممكن بسهولة ترجع (تعمل rollback) للـ version القديم اللي كان شغال كويس. الـ version control بتاعك بيكون حديد.

    • بتفصل بين الشغل اللي لسه بيتعمل والشغل اللي جاهز والشغل اللي عند العميل: ده بيقلل المخاطر جداً.

  • العيوب أو الحاجات اللي ممكن تضايق في Git-Flow (Cons):

    • ممكن تكون معقدة شوية بسبب كتر الـ branches: لو فريقك صغير أو مشروعك بسيط، ممكن تحس إنها "حوار" وخطوات كتير ملهاش لازمة. كتر الـ branches دي (feature, develop, release, hotfix, main) ممكن يزود فرصة حدوث merge conflicts

    • سرعة التطوير والـ release ممكن تبطأ شوية: عشان تطلع feature واحدة للعميل، بتمر برحلة طويلة: من feature branch لـ develop، وبعدين تستنى يتجمعوا كام feature عشان تعمل release branch، وبعدين تختبر، وبعدين تودي على main. اللفة دي ممكن تكون بطيئة لو انت من النوع اللي بيحب يعمل deploy كل يوم أو كل كام ساعة (زي في فلسفة الـ Continuous Integration/Continuous Delivery - CI/CD).

    • محتاجة الفريق كله يكون "على قلب رجل واحد" وملتزم بتطبيقها: لازم كل الناس في الفريق تكون فاهمة الطريقة دي كويس جداً، ومقتنعة بيها، وملتزمة بتطبيقها بحذافيرها. لو كل واحد اشتغل بمزاجه، هتبقى سلطة وهتبوظ الدنيا أكتر ما هتصلحها. محتاجة discipline و team consensus .

  • نقط إضافية كده على الماشي:

    • Git-Flow بتنور أوي في المشاريع اللي ليها release cycle (دورة إصدار) واضحة ومحددة، يعني مثلاً بتطلع version جديدة كل أسبوعين، أو كل شهر. مش مناسبة أوي للمشاريع اللي بتطلع تحديثات طياري كل شوية.

    • مهم أوي إن الـ commit messages (الرسالة اللي بتكتبها مع كل commit) تكون واضحة ومعبرة. لما تيجي تبص في الـ history بتاع أي branch بعد فترة، لازم تفهم كل commit كان بيعمل إيه. متكتبش "fixes" وتمشي!(الموضوع ده بقى شبه محلول حاجة زي vscode بقى فيها AI بيقدر يعملك cmmit message احسن من اللي هتكتبها )

    • فيه أدوات بتساعد على تطبيق Git-Flow بشكل أسهل شوية، زي إضافات لـ Git نفسه أو برامج GUI tools زي SourceTree أو GitKraken اللي بتخليك تشوف الـ branches قدامك كأنها شجرة وتعمل merge و branch بالماوس.

    • GitKraken

    • Source Tree

2. GitHub-Flow:

GitHub-Flow ده يا سيدي جه عشان يبسط الدنيا خالص عن Git-Flow. شال من دماغه حوار الـ release branches الطويلة دي كلها. الفكرة الأساسية بتاعته بتلف حوالين branch واحد بس للتطوير النشط (غالباً بيكون main أو master لو لسه بتستخدم الاسم القديم)، والـ branch ده هو اللي بيتعمل منه deploy مباشرة على الـ production! أيوه زي ما سمعت كده، على طول على الإنتاج.

  • إزاي بقى؟

    1. أي حاجة عايز تعملها (feature جديدة، bug fix) بتاخدلها feature branch جديد من الـ main.

    2. تشتغل على الـ feature branch ده براحتك، تعمل commits، وتعمل push للشغل بتاعك على الـ remote repository (زي GitHub) أول بأول.

    3. لما تحس إن الـ feature بتاعتك جاهزة، بتفتح Pull Request (أو Merge Request لو بتستخدم GitLab مثلاً).

    4. هنا بقى بيحصل الـ discussion والـ code review بين أعضاء الفريق. ممكن حد يسيبلك كومنتات، تعديلات، وهكذا.

    5. لو فيه أي تعديلات مطلوبة، بتعملها على نفس الـ feature branch بتاعك وتعمل push تاني. الـ Pull Request بيتحدث أوتوماتيك.

    6. (اختياري ومهم جداً) ممكن يكون عندك CI (Continuous Integration) شغال، أول ما تفتح Pull Request أو تعمل push جديد، الـ CI server ياخد الكود بتاعك ويعمل build ويشغل tests أوتوماتيك. لو كله أخضر يبقى زي الفل.

    7. لما الـ Pull Request ياخد approval من الفريق والـ CI tests تكون كلها passing، أي حد معاه صلاحية (أو انت نفسك) بيعمل merge للـ Pull Request ده على الـ main branch.

    8. وهنا النقطة الجوهرية: بمجرد ما الـ merge يحصل على main، المفروض إن الكود ده يتعمله deploy على الـ production فوراً (أو في أسرع وقت ممكن). عشان كده لازم الـ main branch يكون دايماً في حالة "deployable" (جاهز للـ deploy).

  • الـ Branches الأساسية:

    • main (أو master): ده الـ branch الوحيد اللي "طويل الأمد" (long-living). هو ده اللي بيمثل الـ production. أي حاجة بتدخل فيه المفروض تكون جاهزة 100% للـ deploy.

    • feature branches (أو topic branches): دي branches "قصيرة الأمد" (short-lived). بتعملها عشان تشتغل على حاجة معينة (feature, bug fix, experiment). بمجرد ما شغلك يخلص ويتعمله merge على main، الـ feature branch ده بيتمسح (أو على الأقل مبقاش له لازمة).

  • ليه سموه GitHub-Flow؟ لأنه الطريقة اللي شركة GitHub نفسها بتستخدمها (أو كانت بتستخدمها بشكل كبير) في تطوير منتجاتها، وهي اللي نشرت الفكرة دي. بتشجع جداً على الـ feedback loops السريعة و asynchronous collaboration، وده بيكون حلو أوي في المشاريع الـ open-source أو الفرق اللي شغالة عن بعد.

  • مميزات GitHub-Flow (Pros):

    • دورات feedback أسرع ودورات إنتاج أقصر: بما إنك بتعمل deploy بسرعة من main، بتقدر تشوف شغلك لايف وتاخد feedback من المستخدمين أو الـ stakeholders بسرعة.

    • مثالي asynchronous work في الفرق الصغيرة: كل واحد شغال على الـ feature branch بتاعه، ولما يخلص بيفتح Pull Request. النقاش بيحصل على الـ Pull Request وكل واحد بيرد في الوقت المناسب ليه.

    • " أسهل بكتير من لخبطة Git-Flow.": مفيش تعقيدات كتير، هو main و feature branches. أي حد جديد في الفريق بيفهم الدنيا بسرعة.

    • بيشجع على الـ Continuous Delivery (CD): فكرة إن main دايماً جاهز للـ deploy بتخلي الـ CD أسهل في التطبيق.

  • عيوب GitHub-Flow (Cons):

    • مجرد ما تعمل merge لـ feature branch ده معناه إنه بقى جاهز للـ production، وده ممكن يدخل bugs لو مفيش اختبارات كافية وعملية CI/CD قوية: دي أخطر نقطة. لو معندكش automated tests محترمة و CI/CD pipeline بتعمل deploy و rollback كويس، ممكن تلبس في الحيط. لازم تكون واثق إن أي حاجة بتدخل main مش هتبوظ الدنيا.

    • الـ branches اللي بتعيش فترة طويلة (long-living branches) ممكن تعقد الدنيا: هو المفروض الـ feature branches تكون قصيرة، بس لو حد خد feature branch وفضل شغال عليه أسابيع أو شهور من غير ما يعمل merge، الـ branch ده بيبعد أوي عن main، وساعتها الـ merge بتاعه ممكن يبقى كابوس ومليان conflicts.

    • صعب تطبيقه على نطاق واسع في الفرق الكبيرة بسبب زيادة احتمالية الـ merge conflicts: لما يكون عندك عشرات المطورين شغالين على نفس الـ main branch (حتى لو عن طريق feature branches)، فرصة إن شغلهم يتعارض مع بعض بتزيد جداً.

    • صعب تدعم كذا release version مختلفة شغالة في نفس الوقت: بما إن main هو الـ production الحالي وبس، لو عايز تعمل support لـ version قديمة وتصلح فيها bugs، هتبقى قصة. GitHub-Flow مش مصمم لكده أصلاً. هو للفرق اللي بتطلع تحديثات سريعة ومتتالية لمنتج واحد.

  • نقط إضافية مهمة:

    • الـ Pull Requests هي أهم خطوة هنا. عشان أي كود جديد يروح للـ main branch ، لازم الأول يتعمله Pull Request. ده بيخلي الفريق كله يراجع الكود (code review) ويتأكد إنه سليم قبل ما يخش على main.

    • الـ Automated Testing ضروري جداً: مينفعش تعتمد على GitHub-Flow لو معندكش شبكة أمان من الـ tests (unit, integration, e2e).

    • الـ Rollbacks لازم تكون سهلة: لو عملت deploy لحاجة بوظت الدنيا، لازم يكون عندك طريقة سهلة وسريعة ترجع للـ version اللي قبلها.

    • مش مناسب لكل المشاريع: لو بتعمل منتج بيطلع منه releases كل فترة طويلة (شهور مثلاً)، أو عندك عملاء كتير كل واحد بـ version مختلفة، أو بتعمل حاجة critical جداً مينفعش فيها أي غلطة (زي برامج بنوك أو طيران)، يمكن GitHub-Flow ميكونش أنسب اختيار.

أمثلة لـ GitHub-Flow:

مثال 1: Feature بسيطة من أولها لآخرها

  • الدنيا بسيطة وجميلة. feature branch، شغل، Pull Request، مراجعة، merge على main، ثم deploy للـ production مباشرة بـ tag جديد.

مثال 2: Bug Fix سريع

  • نفس الـ flow بتاع الـ feature بس لحاجة أصغر وأسرع. الـ merge على main يعني الـ fix بقى لايف.

GitHub-Flow يعتبر الأبسط والأسرع لو فريقك صغير، وشغال على منتج واحد بيطلعله تحديثات باستمرار، وعندك CI/CD و testing قوي. بس لو الشروط دي مش متوفرة، ممكن يسبب مشاكل أكتر ما يحل.

3. GitLab-Flow:

GitLab-Flow ده يا سيدي بيحاول يعمل "توازن" كده، يجمع بين حلاوة البساطة بتاعة GitHub-Flow وبين شوية تنظيم زيادة بس مش بنفس تعقيد Git-Flow. الفكرة الأساسية بتاعته إنه بياخد فكرة إن الـ main branch بتاعك المفروض يكون دايماً جاهز للـ deploy (زي GitHub-Flow)، بس بيضيف عليها حتة مهمة: branches زيادة بتمثل مراحل أو بيئات مختلفة قبل ما الحاجة تطلع للـ production النهائي. يعني ممكن يكون عندك branch للـ staging ، وممكن branch للـ pre-production ، وفي الآخر الـ production branch (أو الـ main نفسه بيكون هو الـ production في بعض الأحيان).

  • إيه القصة هنا؟ الفكرة إنك مش بتعمل deploy من الـ feature branch على طول للـ production زي GitHub-Flow، ولا بتمر باللفة الطويلة بتاعة release branches المنفصلة زي Git-Flow. لأ، هنا أنت بتخلص الـ feature بتاعتك، ممكن تعملها merge على main (أو master لو لسه بتستخدم الاسم القديم). الـ main ده بيكون هو الـ branch الرئيسي بيكون فيه الكود المشارع لاخر اصدار ويكون حصله test واتأكدت انه مستقر. بعد كده، عشان تعمل deploy، بتاخد branch من main وتوديه على بيئة الـ staging. تختبر هناك براحتك. لو كله تمام، ممكن تاخد branch تاني من main (أو تعمل merge من staging لو الدنيا بسيطة) وتوديه لبيئة الـ pre-production (لو عندك). وبعدين في الآخر، الـ production بيتاخد من الـ branch اللي قبله اللي اتجرب وبقى زي الفل.

  • الـ Branches المشهورة في GitLab-Flow :

    • main (أو master): ده الـ branch الأساسي للتطوير، بيكون دايماً فيه الكود اللي المفروض يكون جاهز إنه يطلع للبيئات المختلفة. أي feature جديدة بتخلص، بتتعملها merge هنا. مش بالضرورة يكون هو نفسه اللي بيتعمل منه deploy للـ production مباشرة، لكنه مصدر الـ production.

    • feature branches: زي أي نظام، بتعمل feature branch من main (أو من develop لو قررت تستخدم develop branch وده وارد)، تشتغل عليه، ولما تخلص ترجعه تاني للـ branch اللي طلعت منه (غالباً main).

    • Environment Branches: دي هي الإضافة المميزة هنا.

      • staging branch: بيتعمل deploy منه على بيئة الـ staging.

      • pre-production branch: بيتعمل deploy منه على بيئة قبل الـ production النهائية.

      • production branch: ده الـ branch اللي بيمثل الكود اللي شغال لايف عند العملاء. الـ deploy للـ production بيحصل منه.

      • flow: الشغل بيمشي من main -> staging -> pre-production -> production. يعني بتعمل merge من main لـ staging، وبعدين من staging لـ pre-production (لو موجود)، وبعدين من pre-production لـ production. أو بطريقة تانية، ممكن كل بيئة بتاخد branch نضيف من main بعد ما الـ features اتعملها merge عليه، وتعمل deploy للـ branch ده.

    • release branches : لو عندك releases قديمة لسه بتعملها support، ممكن تعمل release branches منفصلة للإصدارات دي (مثلاً 1-0-stable, 1-1-stable) وتعمل عليها hotfixes لوحدها.

 

  • دورة حياة الشغل (مثال مبسط):

    1. تعمل feature branch من main.

    2. تخلص الـ feature وتعملها merge على main.

    3. (أوتوماتيك أو يدوي) تعمل deploy لـ main على بيئة الـ development أو integration لو موجودة.

    4. لما تكون جاهز، تعمل merge من main إلى staging branch، وده يعمل deploy لبيئة الـ staging.

    5. تختبر على staging. لو فيه bugs، صلحها في feature branch جديد وارجع اعملها merge على main الأول، وبعدين كرر خطوة 4 (ده بيسموه "Upstream first"). متصلحش مباشرة على staging branch عشان متعملش divergence.

    6. لو staging تمام، تعمل merge من staging إلى production branch (أو من main مباشرة لـ production branch لو بتفضل كده)، وده يعمل deploy لبيئة الـ production.

  • مميزات GitLab-Flow (Pros):

    • بتعرف تتعامل كويس مع كذا مرحلة release أو كذا stages قبل الـ production: دي ميزة قوية جداً، بتخليك تختبر الحاجة كويس أوي على مراحل قبل ما توصل للعميل.

    • أبسط من Git-Flow: مفيهاش كتر الـ branches الطويلة الأمد زي develop و release (بالشكل التقليدي بتاع Git-Flow).

    • بتركز على الجودة بطريقة "خفيفة" (lean approach): عندك مراحل واضحة للاختبار (staging, pre-prod)، بس من غير التعقيدات الزيادة. الـ CI/CD (Continuous Integration/Continuous Delivery) بيلعب دور مهم هنا، كل push على branch بيئة معين ممكن يعمل trigger لـ deploy أوتوماتيك.

  • عيوب GitLab-Flow (Cons):

    • التعقيد بيزيد لو بتحاول تدير كذا version مختلفة من المنتج نفسه في نفس الوقت (مش بس stage مختلفة): لو عندك مثلاً v1 و v2 شغالين للعملا وكل واحد له support، هنا ممكن تحتاج تفكر في حلول أعقد شوية، وممكن Git-Flow يكون أنسب في الحالة دي.

    • أكيد أعقد شوية من GitHub-Flow: بسبب وجود الـ environment branches الإضافية دي، فيه خطوات زيادة عن مجرد merge على main و deploy.

  • نقط إضافية كده مهمة (زيادة عن اللي في المقال):

    • مصممة عشان الـ CI/CD: GitLab-Flow بتشتغل زي الحلاوة مع أدوات الـ CI/CD زي GitLab CI (طبعاً!). فكرة إن كل merge على environment branch معين يعمل deploy أوتوماتيك للبيئة دي بتسهل الدنيا جداً.

    • الـ Feedback Loop سريع: لما بتعمل deploy على staging بسرعة، بتقدر تاخد feedback من الـ QA أو حتى من الـ stakeholders بدري بدري.

    • الـ Merge Requests (MRs) أو Pull Requests (PRs) أساسية: كل merge بين الـ branches (خصوصاً لـ main أو للـ environment branches) المفروض يتم عن طريق Merge Request عشان يتعمله code review.

    • الـ Hotfixes: لو حصل bug في production، ممكن تعمل hotfix branch من الـ production branch، تصلحه، تعمل merge تاني للـ production branch (وتعمل deploy فوراً)، وبعدين تعمل merge للـ hotfix ده على main، ومنه لكل الـ environment branches اللي تحت (زي staging). أو ممكن تعمل الـ hotfix على main الأول، وتعمله cherry-pick للـ production branch وتعمل deploy.

  • Important

    لما بتعمل cherry-pick لـ commit معين، إنت كأنك بتقول لـ Git: "أنا عايز آخد التغييرات اللي حصلت في الـ commit ده بالذات (من أي branch تاني) وأطبقها عندي على الـ branch اللي أنا واقف عليه حالياً كـ commit جديد". ده بيكون مفيد جداً لما تكون عايز تنقل إصلاح bug معين أو feature صغيرة من branch لـ branch تاني من غير ما تاخد كل التغييرات اللي حصلت في الـ branch الأولاني. في حالة الـ hotfix، ممكن تصلح الـ bug على main، وبعدين تعمل cherry-pick للـ commit بتاع الإصلاح ده وتوديه للـ production branch عشان يتصلح هناك بسرعة.

     

     

    • المرونة: ممكن تعدل فيها حسب احتياجاتك. بعض الفرق بتستخدم develop branch مع GitLab-Flow (بيبقى develop -> main -> environment branches). وعدد الـ environment branches ممكن يزيد أو يقل.

 

أمثلة عشان الدنيا تنور في GitLab-Flow:

مثال 1: رحلة feature بسيطة من التطوير للإنتاج مروراً بمحطة الـ staging (هنا بنفترض إن main هو الـ branch اللي بنجمع فيه الـ features الجاهزة، وبعدين بننقل منه للبيئات المختلفة زي staging و production.)

 

  • الـ feature الجديدة خلصت وراحت لـ main. بعد كده، عملنا branch جديد اسمه staging من main عشان ننقل عليه الكود ده ونعمله deploy ونجربه كويس. لما اتأكدنا إن كل حاجة شغالة تمام على staging، عملنا branch تاني اسمه production (ممكن ناخده من staging أو من main تاني لو حابين نضمن إننا واخدين نفس النسخة اللي على main بعد ما اتجربت على staging)، ودمجنا فيه الشغل، وده اللي اتعمله deploy للعملاء مع tag جديد للـ version.

مثال 2: سيناريو التعامل مع hotfix في GitLab-Flow (في الحالة دي، هنفترض إن الـ production branch هو اللي ظهرت فيه المشكلة اللي محتاجة تتصلح فوراً.)

  • ظهر bug خطير على الـ production branch (v1.1). عملنا hotfix branch جديد منه مباشرة، صلحنا المشكلة، ودمجنا الإصلاح تاني على production وعملنا deploy للنسخة الجديدة (v1.1.1) للعملاء. بعد كده، لازم ناخد الإصلاح ده ونرجعه لـ main branch عشان ميتنسيش، ومن main بنحدث بيئة الـ staging وباقي البيئات (لو فيه) عشان كلهم يبقوا واخدين نفس الإصلاح.

 

4. Trunk Based Development (TBD):

بص يا كبير، الـ Trunk Based Development أو TBD ده أسلوب شغل كده مختلف عن الباقيين. فكرته إن كل المبرمجين شغالين مع بعض على نفس الـ branch الرئيسي، اللي هو الـ trunk (وده غالباً بيكون الـ main branch بتاعك). القصة كلها إننا عايزين نخلص من حوار الـ branches اللي بتقعد فاتحة كتير دي، زي الـ develop branch بتاع Git-Flow أو حتى الـ feature branches اللي بتطول.

  • إيه الحكاية هنا؟ الهدف كله إن كود كل واحد فينا يتـ integrate (يتدمج) مع الـ trunk الرئيسي ده كل شوية وبسرعة. يعني بدل ما واحد ياخد feature branch ويختفي بيه أسبوعين تلاتة، وبعدين يرجع يلاقي الدنيا مقلوبة و conflicts مالهاش آخر، لأ، في TBD بنقولك قطع الشغلانة الكبيرة لحتت صغيرة أوي (small feature slices أو small changes). كل حتة تخلص، تترمي على طول في الـ trunk. كده الـ trunk بيبقى دايماً فيه آخر شغل من كل الناس، والاختلافات بين كودك وكود غيرك بتبقى حتة صغيرة، فمش هتلاقي conflicts كتير، ولو حصلت بتبقى سهلة.

  • فيه طريقتين للعب بـ TBD، على حسب عدد الناس اللي في التيم:

    1. الطريقة الأولانية: للتيم الصغير (أو التيم اللي قلبه جامد والـ automation عنده حديد)

      • الفكرة: المبرمجين بيعملوا commit دوغري على الـ trunk (اللي هو main). مفيش بقى feature branches ولا وجع دماغ (أو قليل أوي).

      • بس استنى، دي ليها شروط:

        • التيم كله ثقة في بعضه وعارف شغل بعضه.

        • المبرمجين نفسهم تقال وبيعرفوا يكتبوا كود نضيف من أول مرة.

        • عندك نظام automated tests (زي unit tests و integration tests) صاروخ أرض-جو، بيشتغل مع كل commit عشان يلقط أي غلطة في ساعتها.

        • الـ commits نفسها لازم تكون حتة صغيورة قد كده ومركزة على تغيير واحد بس.

      • يوم المبرمج ماشي إزاي هنا:

        1. Update نفسك: قبل ما تبدأ أي حاجة، تسحب آخر تعديلات من الـ trunk (git pull origin main). لازم تكون شغال على أحدث حاجة.

        2. اكتب الكود بتاعك: تعمل التغيير الصغير اللي عليك (حتة feature صغننة، تصلح bug على الماشي).

        3. Test عندك الأول: تشغل الـ unit tests وكل الـ pre-commit checks على جهازك عشان تطمن إن شغلك مكسرش حاجة.

        4. Commit و Push على طول: لو كله تمام، تعمل commit لشغلك ده مباشرة على الـ trunk وترميه على السيرفر فوراً

        5. الـ CI Pipeline تشتغل: أول ما الـ push بتاعك يوصل، الـ Continuous Integration (CI) بيبدأ شغله أوتوماتيك. يعمل build للكود، ويشغل كل الـ automated tests اللي عندك (ممكن تكون أكتر من اللي شغلتها على جهازك).

        6. راقب ولو فيه مشكلة صلحها فوراً: لو الـ CI pipeline عدت بسلام (كل الـ tests نجحت)، يبقى قشطة، الـ trunk لسه نضيف. لو فشلت، يبقى الـ commit بتاعك بوظ الدنيا! هنا لازم تصلح المشكلة دي في ساعتها. ممكن تعمل revert للـ commit بتاعك، أو تصلح الغلطة وتعمل commit جديد بسرعة. هنا مفيش هزار، "لو كسرت الـ build، بتسيب كل اللي في إيدك وتصلحه".

        7. بعد كل commit ناجح على الـ trunk، ممكن الـ CI/CD pipeline تعمل deploy أوتوماتيك للنسخة دي على testing environment أو staging environment عشان تتجرب هناك.

    2. الطريقة الثانية: للتيم الكبير (أو التيم اللي بيحب الـ code review قبل أي حاجة)

      • الفكرة: المبرمجين بيعملوا feature branches، بس بتكون قصيرة جداً جداً (very short-lived). يعني الـ branch ده آخره معاك كام ساعة، أو يوم شغل بالكتير أوي.

      • طب لازمة الـ feature branch القصير ده إيه؟ مش عشان تشتغل لوحدك فترة طويلة، لأ، كل دوره إنه يسمح إن حد من زمايلك يعمل code review على شغلك (عن طريق Pull Request (PR) أو Merge Request (MR)) قبل ما الكود بتاعك يدخل الـ trunk.

      • يوم المبرمج ماشي إزاي هنا:

        1. Update الـ Trunk: زي فوق، تبدأ بسحب آخر نسخة من الـ trunk.

        2. اعمل Branch صغنن: تعمل branch جديد من الـ trunk وتسميه اسم معبر عن الشغلانة الصغيرة اللي هتعملها (مثلاً feature/add-user-avatar أو fix/login-button-issue).

        3. اكتب الكود بتاعك واعمل Commit على الـ branch ده: تعمل شغلك على الـ feature branch ده. ممكن تعمل كذا commit صغير عليه وأنت شغال.

        4. Test عندك الأول: تشغل الـ tests على جهازك.

        5. Push الـ Branch وافتح PR/MR: لما تخلص الحتة الصغيرة دي (في كام ساعة كده)، تعمل push للـ feature branch بتاعك وتفتح Pull Request عشان تطلب إن الكود بتاعك يتعمله merge على الـ trunk.

        6. الـ CI على الـ PR: الـ CI server بيشغل الـ tests على الكود اللي في الـ PR بتاعك عشان يتأكد إنه سليم.

        7. Code Review: واحد أو اتنين من زمايلك في التيم يبصوا على الكود بتاعك بسرعة (بما إن التغييرات حتة صغيرة، المراجعة المفروض تكون سريعة).

        8. صلح لو فيه ملاحظات: لو فيه أي كومنتات، بتعدلها على نفس الـ feature branch بتاعك وتعمل push تاني.

        9. Merge للـ Trunk: لما الـ PR ياخد approval (موافقة) والـ CI يبقى أخضر، بيتعمل merge للـ feature branch على الـ trunk.

        10. امسح الـ Branch: الـ feature branch ده خلاص عمل اللي عليه، بيتمسح على طول.

        11. الـ CI على الـ Trunk تاني: الـ CI بيشتغل مرة تانية على الـ trunk بعد الـ merge عشان يتأكد إن مفيش أي حاجة باظت من عملية الدمج دي.

  • المبادئ الأساسية للـ TBD (دي الخلاصة):

    • Single Shared Branch (Trunk): كله شغال على branch واحد مشترك.

    • Short-Lived Branches (if any): لو هتعمل branches، يبقى عمرها قصير جداً.

    • Frequent Integration: الكود بيتدمج في الـ trunk كل شوية (كذا مرة في اليوم).

    • Small Commits/Slices: الشغل متقطع حتت صغيرة.

    • Comprehensive Automated Testing: لازم tests كتير وأوتوماتيك عشان الـ trunk يفضل نضيف.

    • Continuous Integration (CI): أساسي، كل commit أو merge على الـ trunk بيشغل build و tests.

    • Feature Flags/Toggles: دي حيلة حلوة. لو بتدمج feature لسه مش جاهزة 100% أو مش عايز تظهرها للعميل دلوقتي، بتلفها بـ feature flag. يعني الكود بتاعها موجود في الـ trunk، بس مش متشافة في الـ production إلا لما تقرر تفتح الـ flag ده.

  • إزاي بنعمل Release في TBD؟

    • الـ Release من الـ Trunk دوغري: بما إن الـ trunk (المفروض يعني) دايماً نضيف وجاهز للـ release بسبب الـ CI والـ tests الكتير، ممكن تعمل release منه في أي وقت مزاجك حلو فيه.

      1. تختار الـ Commit: بتنقي commit معين من الـ history بتاع الـ trunk يكون "أخضر" (يعني كل الـ tests بتاعته تمام) ومستقر. غالباً بيكون آخر واحد.

      2. تعمل Tag: بتعلم الـ commit ده بـ tag برقم الـ version (مثلاً v1.0.0, v1.2.5). الـ tag ده هو اللي بيقول "ده الـ release بتاعنا".

      3. Deploy: الـ commit اللي اتعمله tag ده هو اللي بيتاخد وبيتعمله build و deploy على الـ production environment.

    • التعامل مع Hotfixes (لما تحصل مصيبة في release شغال):

      1. الطريقة السريعة (بتاعة TBD ):

        • صلح الـ bug على الـ trunk على طول (سواء بـ commit مباشر أو بـ short-lived branch سريع جداً جداً).

        • الـ CI هيتأكد إن الإصلاح تمام.

        • اعمل tag جديد على الـ commit بتاع الإصلاح ده على الـ trunk (يعني لو كنت على v1.2.5، الإصلاح يخليها v1.2.6).

        • اعمل deploy للـ tag الجديد ده للـ production.

      2. الطريقة التانية (لو الـ trunk سابق بكتير ):

        • اعمل branch جديد من الـ tag بتاع الـ release القديم اللي فيه المشكلة (مثلاً من tag v1.2.5، تعمل branch اسمه hotfix/1.2.5-critical).

        • صلح الـ bug على الـ hotfix branch ده.

        • اعمل tag جديد للـ commit بتاع الإصلاح على الـ hotfix branch ده (مثلاً v1.2.6).

        • اعمل deploy للـ tag الجديد ده (v1.2.6) للـ production.

        • الحتة المهمة هنا: لازم بعد كده تعمل merge (أو cherry-pick) للإصلاح اللي عملته على الـ hotfix branch ده وترجعه تاني للـ trunk الرئيسي، عشان الـ bug ده ميظهرش في الـ releases الجاية اللي هتطلع من الـ trunk.

    • دعم أكتر من Release Version قديمة في نفس الوقت (دي أصعب حتة في TBD): بصراحة، Trunk Based Development مش أحسن حاجة في الدنيا لو شغلك بيجبرك تدعم كذا version قديمة مختلفة وكل واحدة ليها قصة دعم طويلة. هو أصلاً معمول عشان الـ trunk يبقى هو المصدر الوحيد للحقيقة والـ releases تطلع منه زي الصاروخ. بس لو اتزنقت ولازم تعمل كده، الطريقة بتبقى شبه قصة الـ hotfix على release branch قديم:

      1. Release Branches طويلة شوية : لما بتعمل release كبير من الـ trunk (مثلاً v1.0.0) وقررت إنك هتدعمه فترة، ممكن في ساعتها تاخد branch من الـ tag بتاع الـ release ده وتسميه مثلاً release/1.0.x أو 1.0-maintenance.

      2. الـ branch ده المفروض ميدخلش عليه أي features جديدة خالص. هو بس عشان تصلح الـ bugs الجامدة اللي بتظهر في الـ version 1.0.x.

      3. أي hotfix بيتعمل على الـ branch ده زي ما قلنا فوق، وبيطلع منه patch versions جديدة (زي v1.0.1, v1.0.2).

      4. في نفس الوقت، الـ trunk (main) شغال عادي وبيطلع منه releases أجدد (v1.1.0, v2.0.0 ...إلخ).

      5. التحدي هنا: لو فيه bug اتصلح على release/1.0.x وطلع إنه موجود كمان في الكود الأجدد اللي على الـ trunk، لازم حد ياخد الإصلاح ده ويعمله ينقله للـ trunk. ده ممكن يكون عن طريق cherry-pick أو يعيد كتابة الإصلاح. والعكس برضه لو bug اتصلح على الـ trunk وطلع إنه كان موجود من أيام v1.0.x. ده بيزود شوية لخبطة في إدارة الكود، وبيبعد شوية عن فكرة TBD الصافية اللي بتقولك خلي الـ trunk هو كل حاجة. عشان كده، التيمز اللي بتستخدم TBD بتحاول على قد ما تقدر تقلل عدد الـ versions القديمة اللي محتاجة دعم طويل.

  • مميزات Trunk Based Development (Pros):

    • بتخليك تشتغل DevOps صح وتشغل دماغك في الـ unit testing.

    • بتقوي collaboration بين الناس وبتقلل خناقات الـ merge conflicts لأدنى درجة.

    • بتخليك تطلع releases بسرعة الصاروخ (quick releases).

    • الكود بتاعك دايماً "فريش" (up-to-date) وكل الناس شغالة على آخر حاجة.

  • عيوب Trunk Based Development (Cons):

    • عايزة تيم خبرة ويعرف يقطع الـ features حتت صغيرة عشان كله يتدمج بسرعة.

    • معتمدة بشكل كلي على إن الـ CI/CD بتاعك يكون حديد عشان الـ trunk ميبوظش.

    • الـ Code Review ممكن يبقى صعب شوية (خصوصاً لو بتعمل commit مباشر على trunk).

    • صعب تعزل الـ features الكبيرة اللي لسه بتطبخ (وهنا بتيجي أهمية الـ Feature Flags).

 

  • أمثلة لـ Trunk Based Development:

مثال 1: فريق صغير بيعمل commit مباشر على الـ trunk (main)

  • المطورين (A و B) بيعملوا commits صغيرة ومباشرة على الـ trunk (main). كل commit المفروض يكون صغير ومختبر كويس (عن طريق unit tests وبتشغله local و CI بتشتغل على السيرفر). لما يقرروا يعملوا release، بياخدوا آخر commit سليم من الـ trunk ويعملوله tag.

مثال 2: فريق أكبر بيستخدم short-lived feature branches

  • المطورين بيعملوا feature branches بس بتكون "قصيرة العمر جداً" (بتخلص في ساعات أو يوم). بيشتغلوا عليها، يفتحوا Pull Request للمراجعة السريعة، وبمجرد ما تتقبل، بتتعملها merge على طول للـ trunk، والـ feature branch بيتشال. الـ trunk هو اللي بيستقبل كل الشغل ده بسرعة.

Trunk Based Development يعتبر من أكتر الطرق اللي بتدعم الـ Agile والـ DevOps بشكل حقيقي، بس محتاج فريق قوي وأدوات مساعدة (CI/CD, Feature Flags) قوية جداً عشان ينجح. هو مش لكل الناس، بس لما بيتطبق صح، بيكون فعال جداً وبيخلي عملية التطوير سريعة وstable.
